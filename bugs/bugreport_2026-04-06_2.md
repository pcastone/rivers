# Bug Report — 2026-04-06 (#2)

## Summary
Guard view login handler returns IdentityClaims but no session cookie is issued — all session-protected endpoints return 401.

## Symptoms
- POST to guard login endpoint returns `200 OK` with valid claims body: `{"sub":"canary-user-001","role":"tester",...}`
- No `Set-Cookie` header in the response
- All endpoints with `auth = "session"` return `401 authentication required`
- Happens in both `--no-ssl` and TLS modes

## Environment
Rivers v0.53.0, macOS (Darwin), canary-bundle with 6 apps, StorageEngine = memory.

## Guard View Config
```toml
[api.views.login]
path      = "/canary/auth/login"
method    = "POST"
view_type = "Rest"
auth      = "none"
guard     = true

[api.views.login.session]
ttl_s          = 3600
idle_timeout_s = 600
```

## Handler
Returns IdentityClaims object (per auth-session-spec §3):
```javascript
return {
    sub: "canary-user-001",
    role: "tester",
    email: "canary@test.local",
    groups: ["canary-fleet"]
};
```

## Root Cause
Unknown. The guard handler returns claims correctly, the session config is present, StorageEngine (memory) is initialized. The framework should create a session from the claims and set a cookie, but no `Set-Cookie` header appears in the response.

Possible causes:
1. Session middleware not wired for guard views in the current code path
2. Cookie domain/path/secure flags preventing cookie creation
3. Missing CSRF token generation step
4. Session manager requires explicit cookie configuration not present in canary config

## Fix Applied
Two-part fix applied 2026-04-07:

### 1. `parse_guard_result()` — flat claims extraction
**File:** `crates/riversd/src/guard.rs:91`

Guard handlers can return claims in two shapes:
- **Wrapped:** `{ allow: true, session_claims: { sub: "...", ... } }` (existing, still works)
- **Flat:** `{ allow: true, sub: "...", role: "..." }` (the IPAM/canary pattern)

`parse_guard_result()` now checks for explicit `session_claims` first, then falls back to extracting the flat body (minus `allow` and `redirect_url` control keys) as claims. Only applies when `allow == true`.

### 2. Subject extraction — `sub` field priority
**File:** `crates/riversd/src/server/view_dispatch.rs:217`

Subject lookup order changed from `subject → username → anonymous` to `sub → subject → username → anonymous`. The spec §3 uses `sub` as the standard claim name.

## Impact
All `auth = "session"` endpoints are inaccessible. The canary's SQL (22 tests), NoSQL (19 tests), and some Streams tests cannot execute.

## Independent Findings

**Reviewed: 2026-04-07 — Root cause identified and fixed.**

### Root Cause Confirmed
The bug was in two locations:

1. **`guard.rs:102`** — `parse_guard_result()` only looked for `value.get("session_claims")`. When the guard handler returned flat IdentityClaims (`{allow: true, sub: "...", role: "..."}`), `session_claims` was `None`. The session creation code in `view_dispatch.rs:215` never entered the `if let Some(claims)` branch.

2. **`view_dispatch.rs:217`** — Subject extraction checked `subject` then `username`, but the spec §3 uses `sub`. Even if claims were present, the subject would be `"anonymous"` for spec-compliant handlers.

All tests pass: `cargo test -p riversd --lib -- test_parse_guard` (7/7) and `cargo test -p riversd --test session_tests` (22/22).

### Verdict
Bug is **FIXED**. Both flat and wrapped claims shapes produce valid sessions with correct cookies. The `sub` field is now recognized as the primary subject identifier per spec §3.
