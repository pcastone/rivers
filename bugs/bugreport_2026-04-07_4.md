# Bug Report — 2026-04-07 (#4)

## Summary
V8 engine `ctx.dataview()` silently drops host callback error responses — driver errors (forbidden, missing params) returned as "not found" instead of the actual error message.

## Symptoms
- `ctx.dataview("redis_flushdb_trap")` throws `Error: ctx.dataview('redis_flushdb_trap') not found` instead of `Error: forbidden: admin operation 'flushdb' rejected`
- `ctx.dataview("mongo_drop_trap")` same pattern — "not found" instead of "forbidden"
- `ctx.dataview("mongo_delete_by_name", {name: "x"})` throws "not found" instead of `MissingParameter { name: "_filter" }`
- Any DataView that returns an error from the host callback falls through to the generic "not found" throw
- Server log shows the **correct** error message, but JS handler receives the wrong one

## Environment
Rivers v0.53.0, V8 engine cdylib path (host callback dataview dispatch).

## Root Cause

In `crates/rivers-engine-v8/src/execution.rs`, the `dataview_callback()` function only processed the host callback response when the return code was 0 (success):

```rust
// BEFORE (broken)
let result = dv_fn(input.as_ptr(), input.len(), &mut out_ptr, &mut out_len);
if result == 0 && !out_ptr.is_null() && out_len > 0 {
    // Parse and return the result
    ...
}
// Falls through to "not found" throw for ALL non-zero return codes
```

When the host callback returned -10 (driver error) with a valid error response buffer (`{"error": "forbidden: admin operation rejected"}`), the `result == 0` check failed, the buffer was ignored, and execution fell through to:

```rust
let msg = format!("ctx.dataview('{}') not found. Declare in view config: ...", name);
scope.throw_exception(exception);
```

This masked the real error with a misleading "not found" message.

## Fix Applied

**File:** `crates/rivers-engine-v8/src/execution.rs` — `dataview_callback()`

Changed to check for output buffer presence first, then branch on return code:

```rust
// AFTER (fixed)
if !out_ptr.is_null() && out_len > 0 {
    if result == 0 {
        // Success — parse and return
    } else {
        // Error — extract error message from JSON and throw as JS exception
        let err_obj = serde_json::from_str(&json_str);
        let err_msg = err_obj["error"].as_str().unwrap_or(&json_str);
        scope.throw_exception(v8::Exception::error(scope, err_msg));
        return;
    }
} else if result < 0 {
    // Error with no output — throw generic error with host error code
    scope.throw_exception(...);
    return;
}
```

Now JS handlers see the actual driver error:
- `catch(e) { e.message }` → `"forbidden: admin operation 'flushdb' rejected — use application init handler"`
- Admin reject tests can check `errMsg.indexOf("forbidden") >= 0` correctly

## Impact
**Medium** — all `ctx.dataview()` calls that result in driver errors (forbidden operations, missing parameters, connection failures) were misreported as "not found". Handlers using try/catch to detect specific error types (e.g., admin rejection tests) received wrong error messages.
