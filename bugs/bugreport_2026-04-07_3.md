# Bug Report — 2026-04-07 (#3)

## Summary
Static and dynamic plugins loaded simultaneously — cdylib plugins have separate tokio copy, causing "no reactor running" panics that crash riversd.

## Symptoms
- All NoSQL canary tests (19 endpoints) return TIMEOUT
- Server process aborts with: `panicked at tokio-1.50.0/src/net/tcp/stream.rs:161 — there is no reactor running, must be called from the context of a Tokio 1.x runtime`
- Followed by: `fatal runtime error: Rust cannot catch foreign exceptions, aborting`
- Server dies mid-request, all subsequent tests fail with connection refused
- Affects MongoDB, Elasticsearch, CouchDB, and any driver using async TCP (reqwest, tokio::net)
- Redis also affected when using `block_on` path, but worked after switching to `spawn`

## Environment
Rivers v0.53.0, macOS aarch64, default build (`cargo build --release` with `static-plugins` feature enabled). Canary config with `[plugins] dir` pointing to a directory containing cdylib plugin dylibs.

## Root Cause

**The default build compiles plugins twice:**

1. **Static plugins** — `riversd` Cargo.toml has `default = ["static-plugins", ...]` which compiles all 12 plugin crates as `rlib` dependencies linked directly into the `riversd` binary. These share the same tokio runtime as the server.

2. **Dynamic plugins** — `register_all_drivers()` in `crates/riversd/src/server/drivers.rs:51-72` also scans the `[plugins] dir` config path and loads any `.dylib` files via `libloading` (dlopen). These cdylibs each have their **own statically linked copy of tokio**.

When both are registered, the dynamic cdylib version **overwrites** the static rlib version in the `DriverFactory` registry. When the driver's `connect()` or `execute()` runs on a Tokio worker thread, the cdylib's tokio checks its own thread-local reactor context — which was never initialized because only the main binary's tokio runtime is running.

**The panic crosses the cdylib FFI boundary** (the driver trait call enters the cdylib, panics inside, and the unwind crosses back) — this triggers "Rust cannot catch foreign exceptions" and the process aborts.

### Why Redis worked with `spawn` but ES/MongoDB didn't
Initially, `host_dataview_execute` used `rt_handle.block_on()` which ran async code on the V8 engine thread (no reactor). Switching to `rt_handle.spawn()` + mpsc channel moved execution to a Tokio worker thread. This fixed Redis (the builtin Redis driver, not the cdylib plugin) because the static version was still registered first. But ES and MongoDB were being overwritten by the cdylib versions, so the spawn fix didn't help them.

## Fix Applied

### Immediate fix (canary config)
Changed `[plugins] dir` to point to an empty directory, preventing dynamic plugin loading:
```toml
[plugins]
# Static plugins compiled into riversd — don't also load cdylib versions
dir = "/path/to/empty-plugins"
```

### Host callbacks: `block_on` → `spawn` migration
**File:** `crates/riversd/src/engine_loader/host_callbacks.rs`

Changed all three async host callbacks from `rt_handle.block_on()` to `rt_handle.spawn()` + `std::sync::mpsc::channel()`:
- `host_dataview_execute` — DataView queries
- `host_datasource_build` — Direct datasource access
- `host_http_request` — HTTP client requests

This ensures async driver code always runs on a proper Tokio worker thread with reactor context, regardless of whether the caller is a V8 engine thread or other non-Tokio thread.

### Recommended permanent fix (not yet applied)
`register_all_drivers()` should skip dynamic plugin loading when `static-plugins` feature is enabled, or check for duplicate driver names before overwriting:

```rust
#[cfg(feature = "static-plugins")]
{
    // Static plugins already registered — skip dynamic loading
    tracing::info!("static-plugins enabled, skipping dynamic plugin scan");
}
#[cfg(not(feature = "static-plugins"))]
{
    // Load from plugins directory
    let results = load_plugins(plugin_dir, factory);
    ...
}
```

## Impact
**Critical** — crashes the server process, killing all in-flight requests. Any deployment that has both `static-plugins` compiled in AND a populated `[plugins] dir` will hit this on the first NoSQL query.
