# Bug Report — 2026-04-07 (#2)

## Summary
SQL and NoSQL canary DataView queries return null results — drivers load and endpoints respond, but no data is returned from the test cluster.

## Symptoms
- All 12 driver plugins load successfully at startup
- Lockbox resolves 8 credentials (pg, mysql, es, couch, cassandra, ldap, mongo, redis)
- SQL/NoSQL endpoints respond with HTTP 200 and valid test harness JSON
- Every assertion that checks `result_not_null` fails: `"detail": "result=null"`
- Example: `GET /canary-fleet/sql/canary/sql/pg/select` returns:
  ```json
  {"test_id": "SQL-PG-CRUD-SELECT", "passed": false,
   "assertions": [{"id": "result_not_null", "passed": false, "detail": "result type=object"}]}
  ```
- The handler receives a null result from `ctx.dataview()`, indicating the DataView dispatch to the pool manager returns null

## Environment
Rivers v0.53.0, macOS aarch64, dynamic build. Test cluster at 192.168.2.x (all 34 containers running, ES cluster green). Lockbox credentials resolved from `sec/lockbox/rivers.rkeystore`.

## Root Cause
Connection pools are lazy — connections are established on first query, not at startup. The DataView dispatch calls the pool manager, which attempts to acquire a connection from the driver. Possible causes:

1. **Connection timeout** — the pool's `connection_timeout_ms` (default 500ms) may be too short for initial connection establishment to the test cluster over the network
2. **Driver dispatch returns None** — the pool manager may return None when no connection is available, and the DataView engine passes null to the handler instead of waiting or retrying
3. **Credential format mismatch** — lockbox credentials resolved but may not match the expected format for the driver's connection string (e.g., host/port not in the lockbox entry, only username/password)
4. **Missing host/port in datasource config** — the `resources.toml` declares datasources with `lockbox` aliases but the actual `host` and `port` values come from the `app.toml` datasource config. If those are missing or point to wrong addresses, connections fail silently

## Root Cause (updated 2026-04-07)

**Two-part failure:**

### Part 1 (FIXED): HOST_CALLBACKS never set on V8 engine
`load_single_engine()` in `crates/riversd/src/engine_loader/loader.rs` called `_rivers_engine_init()` (no callbacks) instead of `_rivers_engine_init_with_callbacks()`. The V8 engine's `HOST_CALLBACKS` OnceLock was never populated, so `ctx.dataview()` returned null immediately without attempting dispatch.

**Fix:** Changed `load_single_engine()` to prefer `_rivers_engine_init_with_callbacks()` over `_rivers_engine_init()`, passing the HostCallbacks pointer. Confirmed working — `Rivers.log()` and `ctx.data` now function.

### Part 2 (OPEN): DataViewExecutor.execute() returns error for live dispatch
After fixing HOST_CALLBACKS, `ctx.dataview("list_records", {limit: 5})` reaches `host_dataview_execute()` in riversd, which calls `executor.execute()`. The executor either:
1. Can't find the DataView in its registry, or
2. `factory.connect("faker", params)` fails

This happens even for the faker driver (no network needed). The DataView `list_records` is declared in `canary-handlers/app.toml` with `datasource = "canary-faker"`. The datasource config exists with `driver = "faker"`.

**Trace chain:**
- V8 `ctx.dataview()` → `HOST_CALLBACKS.dataview_execute` (now set ✓)
- → `host_dataview_execute()` in `host_callbacks.rs:27` 
- → `HOST_CONTEXT.dataview_executor.execute()` in `host_callbacks.rs:55-60`
- → Returns `Err(...)` → callback returns `-10` → V8 returns null

**Next steps:** Add tracing to `host_dataview_execute()` error path to see the exact error message from `executor.execute()`.

## Fix Applied
Part 1 fixed in `crates/riversd/src/engine_loader/loader.rs` — call `_rivers_engine_init_with_callbacks()` with HostCallbacks pointer.
Part 2 fixed — DataView namespace resolution via `find_by_suffix()` + QueryValue type coercion.

## Impact
All 22 SQL tests and 19 NoSQL tests return `passed: false`. The handlers execute correctly but receive null data from DataView dispatch.
