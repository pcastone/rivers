## 1. Overview

LockBox is Rivers' secret management subsystem. In v1, it is a local age-encrypted keystore file managed by the `lockbox` CLI and decrypted by `riversd` at startup.

Datasources reference secrets via `lockbox://` URIs in `riversd.conf`. At startup, `riversd` decrypts the keystore into host memory. When a datasource connection is established, the host resolves the `lockbox://` URI to the decrypted credential value. The raw credential never enters the ProcessPool isolate — it is held host-side and resolved opaquely.

```
riversd.conf
  credentials_source = "lockbox://postgres/orders-prod"
          │
          ▼
     riversd startup
          │
          ├─ read lockbox.age from configured path
          ├─ decrypt using identity file
          └─ load entries into host-side memory map
                    │
                    ▼
            datasource init
                    │
                    ├─ resolve "postgres/orders-prod" → entry value
                    ├─ establish connection pool using credential
                    └─ credential discarded from resolution context
                              │
                              ▼
                      opaque token issued to ProcessPool
```

## 2. Keystore File Format

### 2.1 Physical format

The keystore is a **TOML document encrypted with age**. The plaintext TOML is never written to disk — the encrypted file is the only on-disk representation.

On-disk: `lockbox.age` — standard age binary format, unarmored.

### 2.2 Plaintext TOML schema

```toml
# Rivers LockBox keystore
# Managed by: lockbox CLI
# Do not edit the plaintext directly — use lockbox commands

[[entry]]
name    = "postgres/orders-prod"
value   = "postgresql://user:password@host:5432/orders"
aliases = ["db/orders", "orders-db"]
created_at = "2026-01-15T10:00:00Z"
updated_at = "2026-01-15T10:00:00Z"

[[entry]]
name    = "anthropic/api_key"
value   = "sk-ant-..."
aliases = ["llm_key", "anthropic_key"]
created_at = "2026-01-15T10:00:00Z"
updated_at = "2026-01-15T10:00:00Z"

[[entry]]
name    = "kafka/prod"
value   = "SASL_SSL://broker:9093?username=rivers&password=secret"
aliases = []
created_at = "2026-01-15T10:00:00Z"
updated_at = "2026-01-15T10:00:00Z"
```

### 2.3 Entry fields

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | string | yes | Canonical name. Used as the lookup key for `lockbox://` URIs. |
| `value` | string | yes | The secret value. Arbitrary string — connection strings, API keys, tokens, passwords. |
| `aliases` | string[] | no | Alternative names that resolve to this entry. May be empty. |
| `created_at` | ISO8601 | yes | Set by `lockbox add`. Never modified. |
| `updated_at` | ISO8601 | yes | Set by `lockbox add` and `lockbox rotate`. |

### 2.4 Name format

Names use forward-slash segments for organizational grouping: `{category}/{identifier}`. The slash is part of the name string — there is no hierarchy enforced. `postgres/orders-prod` and `postgres/billing-prod` are two independent entries that happen to share a prefix.

Names and aliases:
- Case-sensitive
- May contain: alphanumeric, `-`, `_`, `/`, `.`
- May not contain: whitespace, `@`, `:`, `?`, `#`
- Maximum length: 128 characters

### 2.5 Name and alias uniqueness

Names and aliases share a single namespace. No two entries may have the same name, and no entry may have an alias that matches any other entry's name or alias. `lockbox` enforces this at write time. `riversd` enforces it at startup — duplicate names or aliases cause a startup failure.

## 3. Identity File

### 3.1 Format

A standard age X25519 identity file — generated by `lockbox init` or `age-keygen`. Contains the private key used to decrypt the keystore.

```
# created: 2026-01-15T10:00:00Z
# public key: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p
AGE-SECRET-KEY-1QJEPKGMKDUU5TMRLXW0WDXJQLXFQJWZX...
```

Standard age identity file format. `lockbox` and `riversd` use the `age` crate to decrypt — no custom key format.

### 3.2 Location

Configured in `riversd.conf` under `[lockbox]`. Should not be co-located with the keystore file in production — separate paths, separate access controls.

### 3.3 Permissions

`riversd` and `lockbox` enforce file permissions at startup:

| File | Required permissions | Error if violated |
|---|---|---|
| `lockbox.age` | `600` (owner read/write only) | `LockBoxError::InsecureFilePermissions` |
| Identity file | `600` (owner read/write only) | `LockBoxError::InsecureFilePermissions` |

On startup, `riversd` checks permissions before attempting to read either file. Config validation rejects world-readable or group-readable identity files.

## 4. riversd.conf Integration

### 4.1 LockBox config section

```toml
[lockbox]
path          = "/etc/rivers/lockbox.age"        # path to encrypted keystore (required)
identity_file = "/etc/rivers/lockbox.identity"   # path to age identity file (required)
```

Both fields are required when any datasource in the config uses a `lockbox://` URI. If neither field is present and no `lockbox://` URIs are declared, LockBox is inactive and no startup error is emitted.

If `lockbox://` URIs are declared but `[lockbox]` is not configured, `riversd` fails at startup:

```
RiversError::Config: lockbox.path and lockbox.identity_file are required 
when credentials_source uses lockbox:// URIs
```

### 4.2 Example riversd.conf

```toml
[base]
name = "orders-api"

[lockbox]
path          = "/etc/rivers/lockbox.age"
identity_file = "/etc/rivers/lockbox.identity"

[data.datasources.primary_db]
driver             = "postgresql"
host               = "db.internal"
port               = 5432
database           = "orders"
credentials_source = "lockbox://postgres/orders-prod"

[data.datasources.anthropic]
driver             = "http"
base_url           = "https://api.anthropic.com"
auth               = "bearer"
credentials_source = "lockbox://anthropic/api_key"

[data.datasources.cache]
driver             = "redis"
host               = "cache.internal"
port               = 6379
credentials_source = "lockbox://redis/prod"
```

## 5. URI Resolution

### 5.1 URI format

```
lockbox://{name}
```

The `{name}` segment is matched against entry names and aliases in the keystore. The match is exact and case-sensitive.

Examples:
```
lockbox://postgres/orders-prod     → looks up name "postgres/orders-prod"
lockbox://orders-db                → looks up alias "orders-db" (resolves to same entry)
lockbox://anthropic/api_key        → looks up name "anthropic/api_key"
lockbox://llm_key                  → looks up alias "llm_key" (resolves to same entry)
```

### 5.2 Resolution order

1. Match against all entry `name` fields — exact string match
2. If no name match, match against all `aliases` fields — exact string match
3. If no match — `LockBoxError::EntryNotFound` — `riversd` startup fails

### 5.3 Resolution at startup vs runtime

All `lockbox://` URIs declared in `riversd.conf` are resolved at **startup**. `riversd` walks the full datasource config, collects all `lockbox://` URIs, and resolves them against the decrypted keystore before any connection pools are initialized. If any URI fails to resolve, startup fails with a clear error identifying the unresolved URI and the datasource that referenced it.

```
RiversError::LockBox: failed to resolve "lockbox://postgres/staging" 
  referenced by datasource "primary_db"
  — no entry with name or alias "postgres/staging" in keystore
```

This means credential availability is validated at startup, not at first request. A misconfigured credential fails fast, not silently at runtime.

### 5.4 In-memory store

After decryption and resolution, the full keystore is held in a host-side `HashMap<String, SecretEntry>` keyed by both names and aliases for O(1) lookup. The plaintext TOML and the age-decrypted bytes are zeroized immediately after the map is populated — only the resolved values remain in memory for the duration of the process.

The in-memory store is not accessible from handler code. There is no `Rivers.lockbox` API in v1 — credentials are a datasource concern only.

## 6. lockbox CLI

### 6.1 Overview

`lockbox` is a standalone binary that ships alongside `riversd`. It manages the age-encrypted keystore file. All commands that modify the keystore read the existing file, decrypt, apply the change, re-encrypt, and write back atomically (write to `.tmp`, then rename).

`lockbox` reads the identity file path from one of three sources in priority order:
1. `--identity <path>` flag
2. `RIVERS_LOCKBOX_IDENTITY` environment variable
3. `[lockbox].identity_file` from `riversd.conf` (if `--config` flag provided)

### 6.2 Commands

#### `lockbox init`

Generate a new age identity file and an empty encrypted keystore.

```
lockbox init --identity <path> --keystore <path>
```

```
$ lockbox init --identity /etc/rivers/lockbox.identity --keystore /etc/rivers/lockbox.age
Generated identity file: /etc/rivers/lockbox.identity
  Public key: age1ql3z7hjy54pw3hyww5ayyfg7zqgvc7w3j2elw8zmrj2kg5sfn9aqmcac8p
Generated keystore:      /etc/rivers/lockbox.age  (0 entries)
Permissions set to 600 on both files.
```

If either file already exists, `lockbox init` refuses and exits non-zero. Use `--force` to overwrite.

---

#### `lockbox add`

Add a new entry. Prompts for value on stdin (not as a flag — prevents value appearing in shell history or process list).

```
lockbox add <name> [--alias <alias>]... [--identity <path>] [--keystore <path>]
```

```
$ lockbox add anthropic/api_key --alias llm_key --alias anthropic_key
Enter value for "anthropic/api_key": [hidden]
Confirm value: [hidden]
Added entry "anthropic/api_key" with aliases: llm_key, anthropic_key
```

Exits non-zero if `name` or any alias already exists in the keystore.

---

#### `lockbox alias`

Add one or more aliases to an existing entry.

```
lockbox alias <name> <alias> [<alias>...]
```

```
$ lockbox alias anthropic/api_key claude_key
Added alias "claude_key" → "anthropic/api_key"
```

Exits non-zero if the alias already exists in the keystore under any entry.

---

#### `lockbox unalias`

Remove an alias from an entry.

```
lockbox unalias <alias>
```

```
$ lockbox unalias claude_key
Removed alias "claude_key" from entry "anthropic/api_key"
```

---

#### `lockbox list`

List all entries — names and aliases. Values are never printed.

```
lockbox list [--identity <path>] [--keystore <path>]
```

```
$ lockbox list
ENTRIES (3)

  anthropic/api_key
    aliases: llm_key, anthropic_key
    created: 2026-01-15  updated: 2026-01-15

  postgres/orders-prod
    aliases: db/orders, orders-db
    created: 2026-01-15  updated: 2026-01-15

  redis/prod
    aliases: (none)
    created: 2026-01-15  updated: 2026-01-15
```

---

#### `lockbox export`

Decrypt and print an entry's value to stdout. Intended for scripting and debugging. Use with care — value is printed in plaintext.

```
lockbox export <name-or-alias> [--identity <path>] [--keystore <path>]
```

```
$ lockbox export anthropic/api_key
sk-ant-api03-...
```

`lockbox export` emits a warning to stderr when stdout is a TTY:

```
Warning: printing secret value to terminal. Pipe to a file or command to suppress this warning.
```

When stdout is a pipe (`lockbox export anthropic/api_key | some-command`), the warning is suppressed.

---

#### `lockbox rotate`

Replace an entry's value. Aliases and name are preserved. Prompts for new value on stdin.

```
lockbox rotate <name-or-alias> [--identity <path>] [--keystore <path>]
```

```
$ lockbox rotate anthropic/api_key
Enter new value for "anthropic/api_key": [hidden]
Confirm new value: [hidden]
Rotated entry "anthropic/api_key"
  updated_at: 2026-03-01T10:00:00Z
Note: riversd must be restarted to use the new credential.
```

`lockbox rotate` always prints the restart reminder — credential rotation takes effect on the next `riversd` startup.

---

#### `lockbox remove`

Remove an entry and all its aliases.

```
lockbox remove <name> [--identity <path>] [--keystore <path>]
```

```
$ lockbox remove postgres/orders-prod
Remove entry "postgres/orders-prod" and aliases [db/orders, orders-db]? [y/N] y
Removed.
```

Requires confirmation unless `--yes` flag is provided. Exits non-zero if name not found.

---

#### `lockbox rekey`

Re-encrypt the entire keystore with a new identity file. Used for identity rotation.

```
lockbox rekey --new-identity <path> [--identity <path>] [--keystore <path>]
```

```
$ lockbox rekey --new-identity /etc/rivers/lockbox.identity.new
Decrypting with current identity...
Re-encrypting with new identity...
Keystore rekeyed. 3 entries preserved.
Update riversd.conf [lockbox].identity_file to point to the new identity file.
Restart riversd to apply.
```

The old identity file is not deleted — the operator is responsible for retiring it. `lockbox rekey` prints the old identity file path as a reminder.

---

#### `lockbox verify`

Verify the keystore is well-formed and all entries are readable with the configured identity. Non-destructive — no writes.

```
lockbox verify [--identity <path>] [--keystore <path>]
```

```
$ lockbox verify
Decrypting keystore...
Validating 3 entries...
  ✓ anthropic/api_key (aliases: llm_key, anthropic_key)
  ✓ postgres/orders-prod (aliases: db/orders, orders-db)
  ✓ redis/prod (aliases: none)
No duplicate names or aliases found.
Keystore is valid.
```

Exits non-zero on any error — permission issue, decryption failure, duplicate names, malformed entries.

---

### 6.3 Exit codes

| Code | Meaning |
|---|---|
| 0 | Success |
| 1 | Entry not found |
| 2 | Entry already exists (add conflict) |
| 3 | Decryption failed (wrong identity, corrupted file) |
| 4 | Permission error (file permissions, read/write failure) |
| 5 | Keystore validation error (duplicate names, malformed TOML) |
| 6 | User cancelled (confirmation prompt declined) |

## 7. riversd Startup Resolution

### 7.1 Startup sequence

LockBox resolution occurs during `riversd` startup, after config is parsed and before any datasource connections are initialized:

```
riversd startup
    │
    ├─ 1. parse riversd.conf
    ├─ 2. collect all lockbox:// URIs from datasource configs
    │
    ├─ 3. [lockbox] section present?
    │       NO and URIs exist → startup failure: LockBoxConfig missing
    │       NO and no URIs → LockBox inactive, continue
    │       YES → continue
    │
    ├─ 4. check file permissions on lockbox.age and identity_file
    │       failure → LockBoxError::InsecureFilePermissions
    │
    ├─ 5. read identity file
    │       failure → LockBoxError::IdentityReadFailed
    │
    ├─ 6. decrypt lockbox.age with identity
    │       failure → LockBoxError::DecryptionFailed
    │
    ├─ 7. parse decrypted TOML
    │       failure → LockBoxError::MalformedKeystore
    │
    ├─ 8. validate keystore (no duplicate names/aliases)
    │       failure → LockBoxError::DuplicateEntry
    │
    ├─ 9. build in-memory name+alias → value map
    │
    ├─ 10. resolve all lockbox:// URIs from step 2
    │        any unresolved → LockBoxError::EntryNotFound (with datasource context)
    │
    ├─ 11. zeroize plaintext TOML bytes and decrypted age bytes
    │
    └─ 12. initialize datasource connection pools using resolved credentials
```

### 7.2 Failure behavior

Any failure in steps 3–10 causes immediate startup failure. `riversd` exits non-zero with a structured error message identifying the failure type, file path, and (where applicable) the unresolved URI and datasource name. Partial startup is not permitted — `riversd` does not serve traffic with unresolved credentials.

### 7.3 Runtime credential access

After startup, resolved credentials are accessed by the datasource connection pool manager via an internal `LockBoxResolver::get(uri)` call. This is host-side Rust code only — not accessible from ProcessPool isolates. The resolver returns a reference to the in-memory value. The value is not copied per-call.

### 7.4 Credential reload

In v1, credentials are loaded once at startup. There is no hot-reload. After `lockbox rotate`, `riversd` must be restarted to pick up the new value. The `lockbox rotate` command always prints this reminder.

## 8. Datasource Integration

### 8.1 credentials_source field

All driver datasource configs accept a `credentials_source` field. In v1, the only valid value format is `lockbox://{name-or-alias}`.

```toml
[data.datasources.primary_db]
driver             = "postgresql"
host               = "db.internal"
port               = 5432
database           = "orders"
credentials_source = "lockbox://postgres/orders-prod"
```

### 8.2 Credential value semantics

The `value` field in a keystore entry is a raw string. The driver is responsible for interpreting the value. Drivers expect the following formats:

| Driver | Expected credential format |
|---|---|
| `postgresql` | Full connection string: `postgresql://user:pass@host:port/db` or `user:password` |
| `mysql` | Full connection string: `mysql://user:pass@host:port/db` or `user:password` |
| `redis` | Password string or full `redis://:password@host:port` |
| `mongodb` | Full URI: `mongodb://user:pass@host:port` |
| `http` (bearer) | Token string — passed as `Authorization: Bearer {value}` |
| `http` (api_key) | Key string — passed as configured `auth_header: {value}` |
| `http` (basic) | `username:password` — base64-encoded by driver |
| `kafka` | `username:password` — used for SASL authentication |
| `elasticsearch` | `username:password` or API key string |

Driver-specific credential format is documented in the driver spec. If the credential format is invalid for the driver, the datasource connection pool fails to initialize at startup — not at first query.

### 8.3 Credential-free datasources

`credentials_source` is optional. Datasources that do not require authentication omit the field entirely:

```toml
[data.datasources.public_api]
driver   = "http"
base_url = "https://api.public-service.example.com"
# no credentials_source — unauthenticated
```

## 9. Security Model

### 9.1 Threat model

| Threat | Mitigation |
|---|---|
| Keystore file read by unauthorized user | `600` permissions enforced at startup and by `lockbox` on write |
| Identity file read by unauthorized user | `600` permissions enforced at startup |
| Credential value in shell history | `lockbox add` and `lockbox rotate` read value from stdin, never as a CLI argument |
| Credential value in process list | Same — stdin only, never `--value` flag |
| Credential leaking into ProcessPool isolate | Host-side resolution only. No `Rivers.lockbox` API in v1. Opaque token model. |
| Plaintext keystore left in memory after parsing | Plaintext TOML and age-decrypted bytes zeroized after in-memory map is built |
| Credential in log output | `riversd` never logs credential values. LockBox resolution errors log the URI and datasource name only. |
| Keystore corruption | `lockbox verify` — run after any manual file operation |

### 9.2 What LockBox does not protect against

- **Root access** — a root user can read the identity file and decrypt the keystore. LockBox protects against unprivileged access, not root compromise.
- **Process memory inspection** — resolved credential values are held in process memory. A sufficiently privileged attacker with access to the `riversd` process memory can extract them. This is an inherent limitation of any in-process secret store.
- **Credential rotation propagation** — in v1, rotation requires a restart. There is no zero-downtime rotation path. v2 RPS addresses this.

### 9.3 Reserved fields

Per AMD-9, `session:`, `csrf:`, `poll:`, and `rivers:` are reserved StorageEngine key prefixes. `lockbox://` is a reserved URI scheme in `credentials_source`. These are distinct namespaces — LockBox entry names do not interact with StorageEngine key prefixes.

## 10. Validation Rules

Enforced at `riversd` startup and by `lockbox` at write time:

| Rule | Error |
|---|---|
| `[lockbox]` absent, `lockbox://` URIs present | `lockbox config required when credentials_source uses lockbox:// URIs` |
| `lockbox.path` file not found | `LockBoxError::KeystoreNotFound: {path}` |
| `lockbox.identity_file` not found | `LockBoxError::IdentityNotFound: {path}` |
| `lockbox.age` permissions not `600` | `LockBoxError::InsecureFilePermissions: {path} — expected 600` |
| `lockbox.identity` permissions not `600` | `LockBoxError::InsecureFilePermissions: {path} — expected 600` |
| Decryption failure | `LockBoxError::DecryptionFailed — check identity file matches keystore` |
| Duplicate entry name or alias | `LockBoxError::DuplicateEntry: "{name}" appears in multiple entries` |
| Unresolved `lockbox://` URI | `LockBoxError::EntryNotFound: "{uri}" referenced by datasource "{name}"` |
| Entry name contains invalid characters | `LockBoxError::InvalidEntryName: "{name}" — names may only contain [a-zA-Z0-9_/.-]` |
| Entry name exceeds 128 characters | `LockBoxError::InvalidEntryName: "{name}" — maximum length 128 characters` |
| `lockbox add` with existing name | `Entry "{name}" already exists — use lockbox rotate to update the value` |

## 11. v2 Upgrade Path

The `lockbox://` URI scheme is stable. No datasource config changes when upgrading from v1 local LockBox to v2 RPS secret broker.

In v2, the `[lockbox]` section gains an optional `backend` field:

```toml
[lockbox]
backend       = "rps"                            # v2 — remote via RPS
path          = "/etc/rivers/lockbox.age"        # v1 fallback if RPS unavailable
identity_file = "/etc/rivers/lockbox.identity"
```

When `backend = "rps"`, `riversd` fetches credentials from RPS at startup instead of from the local file. The local file remains as a fallback for DR scenarios. The `lockbox://` URIs in datasource configs are unchanged.

Apps that worked with v1 local LockBox work identically with v2 RPS — no code changes, no config changes beyond adding `backend = "rps"`.

## 12. Examples

### 12.1 Initial setup

```bash
# Generate identity file and empty keystore
lockbox init \
  --identity /etc/rivers/lockbox.identity \
  --keystore /etc/rivers/lockbox.age

# Add database credentials
lockbox add postgres/orders-prod \
  --alias orders-db \
  --alias db/orders \
  --keystore /etc/rivers/lockbox.age \
  --identity /etc/rivers/lockbox.identity
# Enter value for "postgres/orders-prod": [hidden]
# Confirm value: [hidden]

# Add API key
lockbox add anthropic/api_key \
  --alias llm_key \
  --keystore /etc/rivers/lockbox.age \
  --identity /etc/rivers/lockbox.identity
# Enter value for "anthropic/api_key": [hidden]
# Confirm value: [hidden]

# Verify keystore
lockbox verify \
  --keystore /etc/rivers/lockbox.age \
  --identity /etc/rivers/lockbox.identity
```

### 12.2 Credential rotation

```bash
# Rotate an API key after it has been cycled upstream
lockbox rotate anthropic/api_key \
  --keystore /etc/rivers/lockbox.age \
  --identity /etc/rivers/lockbox.identity
# Enter new value for "anthropic/api_key": [hidden]
# Confirm new value: [hidden]
# Rotated entry "anthropic/api_key"
# Note: riversd must be restarted to use the new credential.

# Restart riversd
systemctl restart riversd
```

### 12.3 Using RIVERS_LOCKBOX_IDENTITY env var

```bash
export RIVERS_LOCKBOX_IDENTITY=/etc/rivers/lockbox.identity

# All lockbox commands pick up identity from env — no --identity flag needed
lockbox list --keystore /etc/rivers/lockbox.age
lockbox add redis/prod --keystore /etc/rivers/lockbox.age
```
