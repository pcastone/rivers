## 1. Overview

LockBox is Rivers' secret store abstraction. In v1 (single-node, no RPS), secrets live in a local Age-encrypted keystore file. The file is decrypted once at startup, held in memory, and never written to disk in plaintext.

Handler code references secrets by name using the `lockbox://` URI scheme. The host resolves the name to the secret value using an opaque token — the raw value never enters the isolate.

The keystore file is a first-class operational artifact — versioned, backed up, and managed via the `rivers lockbox` CLI. It is not an ad-hoc `.env` file.

---

## 2. File Format

### Outer envelope

The keystore file is an [Age](https://age-encryption.org)-encrypted binary file. The file extension is `.rkeystore`.

```
riversd.rkeystore
```

The encrypted payload is a TOML document. Age handles encryption, authentication, and key management. Rivers does not implement its own crypto envelope.

### Plaintext payload (inside Age envelope)

```toml
# Rivers Keystore — plaintext inside Age envelope
# Do not edit this file directly. Use: rivers lockbox <command>

version = 1

[[entries]]
name    = "anthropic_api_key"
value   = "sk-ant-..."
type    = "string"
aliases = ["anthropic_key", "llm_key"]
created = "2026-03-01T00:00:00Z"
updated = "2026-03-01T00:00:00Z"

[[entries]]
name    = "postgres_password"
value   = "hunter2"
type    = "string"
aliases = []
created = "2026-03-01T00:00:00Z"
updated = "2026-03-01T00:00:00Z"

[[entries]]
name    = "jwt_signing_key"
value   = "base64url:..."
type    = "base64url"
aliases = ["jwt_key", "signing_key"]
created = "2026-03-01T00:00:00Z"
updated = "2026-03-01T00:00:00Z"
```

### File on disk

```
# Encrypted — binary Age format
# Not human-readable outside of CLI tooling
$ file riversd.rkeystore
riversd.rkeystore: Age encrypted file
```

## 3. Entry Model

Each entry has:

| Field | Required | Description |
|---|---|---|
| `name` | Yes | Primary identifier. Unique within keystore. Used in `lockbox://name` references. |
| `value` | Yes | The secret value. Plaintext string inside the encrypted envelope. |
| `type` | Yes | Value type hint for host-side handling. |
| `aliases` | No | Zero or more alternative names that resolve to this entry. |
| `created` | Yes | ISO 8601 UTC timestamp. Set by CLI at creation. |
| `updated` | Yes | ISO 8601 UTC timestamp. Updated by CLI on value change. |

### Value types

| Type | Description | Delivery to host |
|---|---|---|
| `string` | Arbitrary string — API keys, passwords, tokens | Raw string |
| `base64url` | Base64url-encoded binary — signing keys, symmetric keys | Decoded bytes |
| `pem` | PEM-encoded certificate or private key | Raw PEM string |
| `json` | JSON object — structured credentials (OAuth client secrets, service account keys) | Parsed object |

Type determines how the host delivers the resolved value to the datasource driver. The CodeComponent isolate never receives the raw value regardless of type.

### Naming rules

- Name must match `[a-z][a-z0-9_]*` — lowercase alphanumeric and underscore, starting with a letter
- Maximum length: 128 characters
- Names and aliases share the same namespace — an alias cannot duplicate any name or alias in the keystore
- Names are immutable after creation — use `rivers lockbox alias` to add an alias, `rivers lockbox rotate` to update the value

## 4. Alias Resolution

An alias is an alternative name for an entry. Aliases allow:

- Shorthand references (`anthropic_key` instead of `anthropic_api_key`)
- Stable references across key rotation (`llm_key` always points to the current LLM API key regardless of which provider is active)
- Environment-neutral references (`db_password` in config, actual entry varies per environment)

Resolution is flat — aliases resolve directly to entries, not to other aliases. Alias chains are not supported. If an alias points to another alias, CLI validation rejects it at write time.

### Resolution order

1. Exact match on `name`
2. Exact match on any `alias`
3. Not found — startup fails (see §7)

Both steps happen against the in-memory resolved keystore. Resolution is O(1) via hash map built at startup.

### Config reference syntax

```toml
# In datasource config — either name or alias works
[data.datasources.openai]
driver      = "http"
base_url    = "https://api.openai.com"
auth        = "bearer"
credentials = "lockbox://openai_api_key"   # by name

[data.datasources.anthropic]
driver      = "http"
base_url    = "https://api.anthropic.com"
auth        = "bearer"
credentials = "lockbox://llm_key"          # by alias
```

## 5. riversd.conf Integration

The keystore file location is declared in `riversd.conf`:

```toml
[lockbox]
path            = "/etc/rivers/riversd.rkeystore"   # required if any lockbox:// reference exists
key_source      = "env"                              # "env" | "file" | "agent"
key_env_var     = "RIVERS_LOCKBOX_KEY"              # used when key_source = "env"
key_file        = "/etc/rivers/lockbox.key"         # used when key_source = "file"
```

If no `lockbox://` reference exists anywhere in the config, the `[lockbox]` section is optional and the keystore file is not loaded.

If any `lockbox://` reference exists and `[lockbox]` is not configured, startup fails:

```
RiversError::Config: lockbox reference found but [lockbox] is not configured
  → add [lockbox] section to riversd.conf
```

### Path resolution

`path` is an absolute path. Relative paths are rejected at config validation — keystore location must be unambiguous regardless of working directory.

```
RiversError::Config: lockbox.path must be an absolute path, got "riversd.rkeystore"
```

## 6. Decryption Key Sources

The Age identity (decryption key) is never stored alongside the keystore file. Three sources are supported:

### `key_source = "env"` (default for development)

```toml
[lockbox]
key_source  = "env"
key_env_var = "RIVERS_LOCKBOX_KEY"   # default var name
```

`RIVERS_LOCKBOX_KEY` contains the Age identity string (X25519 private key in Bech32 encoding, `AGE-SECRET-KEY-1...`). Rivers reads it at startup, decrypts the keystore, then clears the variable from the process environment.

### `key_source = "file"` (for deployments with secrets management)

```toml
[lockbox]
key_source = "file"
key_file   = "/etc/rivers/lockbox.key"   # absolute path, Age identity file
```

The identity file contains the Age secret key. File permissions are validated at startup — if the file is group- or world-readable, startup fails:

```
RiversError::Security: lockbox key file /etc/rivers/lockbox.key has insecure permissions (mode 0644)
  → chmod 0600 /etc/rivers/lockbox.key
```

### `key_source = "agent"` (for SSH agent / hardware key integration)

```toml
[lockbox]
key_source     = "agent"
agent_socket   = "/run/user/1000/gnupg/S.gpg-agent.ssh"  # SSH_AUTH_SOCK
recipient_file = "/etc/rivers/lockbox.pub"                 # Age recipient (public key)
```

Rivers uses the SSH agent via `age-ssh` plugin. The keystore is encrypted to an SSH key held in the agent. Useful for hardware security keys (YubiKey, etc.) or when the operator requires physical presence for decryption.

## 7. Startup Validation

LockBox validation runs as part of `riversd` startup, before any request is served.

### Validation sequence

```
1. Load riversd.conf [lockbox] section
2. Check keystore file exists and is readable
3. Check key source is available (env var set / key file exists and readable / agent responding)
4. Decrypt keystore file → plaintext TOML
5. Parse TOML → in-memory entry map
6. Validate entry schema (required fields, naming rules, no duplicate names/aliases)
7. Build resolution map: name → entry, alias → entry
8. Validate all lockbox:// references in riversd.conf resolve to known entries
9. Validate all lockbox:// references in datasource configs resolve to known entries
10. Clear plaintext TOML from memory (only resolution map retained)
11. ✓ LockBox ready
```

### Fail-fast behavior

Rivers does not start if any validation step fails. Partial startup with missing secrets is not permitted — a partially configured server is a misconfigured server.

### Error examples

```
RiversError::LockBox: keystore file not found: /etc/rivers/riversd.rkeystore
RiversError::LockBox: decryption failed — wrong key or corrupted file
RiversError::LockBox: unresolved reference "lockbox://missing_key" in datasource "openai"
RiversError::LockBox: duplicate alias "api_key" — already used by entry "anthropic_api_key"
RiversError::LockBox: entry name "2bad" is invalid — must match [a-z][a-z0-9_]*
```

## 8. Rivers CLI Tooling

All keystore operations go through the `rivers lockbox` CLI. The keystore file is never edited directly.

The CLI requires the same key source configuration as `riversd` — it reads `[lockbox]` from `riversd.conf` or accepts `--config` to specify an alternate config file.

### Command reference

#### Create a new keystore

```bash
rivers lockbox init \
    --path /etc/rivers/riversd.rkeystore \
    --key-source env
# Generates a new Age keypair
# Prints: AGE-SECRET-KEY-1... (store this in RIVERS_LOCKBOX_KEY)
# Prints: age1... (recipient public key — used for re-encryption)
# Creates empty keystore at --path
```

#### Add an entry

```bash
rivers lockbox add \
    --name anthropic_api_key \
    --type string \
    --alias anthropic_key \
    --alias llm_key
# Prompts for value (hidden input — not via argument to avoid shell history)
# Value: ****
# ✓ Added "anthropic_api_key" with 2 aliases
```

#### List entries

```bash
rivers lockbox list
# NAME                  TYPE      ALIASES                    UPDATED
# anthropic_api_key     string    anthropic_key, llm_key     2026-03-01
# postgres_password     string    (none)                     2026-03-01
# jwt_signing_key       base64url jwt_key, signing_key       2026-03-01
```

Values are never printed by `list`. Use `rivers lockbox show --name` with explicit confirmation.

#### Show entry (with confirmation)

```bash
rivers lockbox show --name anthropic_api_key
# WARNING: This will display a secret value. Continue? [y/N] y
# name:    anthropic_api_key
# type:    string
# aliases: anthropic_key, llm_key
# value:   sk-ant-...
```

#### Rotate a value (update secret, preserve name and aliases)

```bash
rivers lockbox rotate --name anthropic_api_key
# Current value will be replaced. Continue? [y/N] y
# New value: ****
# ✓ Rotated "anthropic_api_key" — updated timestamp set
# → No config changes required (lockbox:// references unchanged)
```

#### Add an alias to an existing entry

```bash
rivers lockbox alias \
    --name anthropic_api_key \
    --alias new_alias
# ✓ Alias "new_alias" added to "anthropic_api_key"
```

#### Remove an alias

```bash
rivers lockbox unalias \
    --name anthropic_api_key \
    --alias old_alias
# ✓ Alias "old_alias" removed from "anthropic_api_key"
```

#### Remove an entry

```bash
rivers lockbox remove --name anthropic_api_key
# WARNING: Removing "anthropic_api_key" will break any lockbox:// references to this entry.
# Aliases removed: anthropic_key, llm_key
# Type entry name to confirm: anthropic_api_key
# ✓ Removed
```

#### Validate keystore against config

```bash
rivers lockbox validate --config /etc/rivers/riversd.conf
# ✓ Keystore decrypted successfully
# ✓ 3 entries loaded
# ✓ All lockbox:// references resolve
# ✓ No duplicate names or aliases
```

#### Re-encrypt with a new key

```bash
rivers lockbox rekey
# Generates new Age keypair
# Re-encrypts all entries under new key
# Prints new AGE-SECRET-KEY-1...
# Old key is no longer valid
```

## 9. Handler Integration

Handler code does not interact with LockBox directly. Credentials are a datasource concern — they are resolved at the host layer and delivered to the driver. The isolate receives an opaque token for the datasource, not the credential value.

```typescript
// Handler — no lockbox reference, no credential handling
export async function generate(
    req: Rivers.Request
): AsyncGenerator<any> {
    // "anthropic" datasource has credentials resolved host-side from lockbox
    for await (const chunk of Rivers.view.stream("generate_completion", {
        prompt: req.body.prompt,
    })) {
        yield chunk;
    }
}
```

```toml
# Datasource config — lockbox reference is here, not in handler
[data.datasources.anthropic]
driver      = "http"
base_url    = "https://api.anthropic.com"
auth        = "bearer"
credentials = "lockbox://anthropic_api_key"
```

The `lockbox://` URI appears only in `riversd.conf` — in datasource configs, session signing key config, and other infrastructure config. It never appears in TypeScript/WASM handler source.

## 10. Security Properties

### At rest

- Keystore file is Age-encrypted (X25519 + ChaCha20-Poly1305 AEAD)
- Plaintext never written to disk — decrypted in memory at startup only
- File permissions validated — insecure permissions fail startup

### In memory

- Plaintext TOML cleared after parsing — only the resolution map (name → resolved value) is retained
- Resolution map lives in the `riversd` host process — not in any isolate
- Isolates hold opaque datasource tokens, not credential values

### In transit

- Credentials never logged — the ad-hoc HTTP call warning (AMD-1) logs the host, not query params or headers that may contain credentials
- Credentials never appear in error messages — `LockBox: decryption failed` not `LockBox: failed to decrypt key sk-ant-...`
- `rivers lockbox show` requires explicit terminal confirmation — not scriptable without `--yes` flag (which also emits a warning log)

### Key rotation

- `rivers lockbox rotate` updates the value, preserves all `lockbox://` references
- `rivers lockbox rekey` re-encrypts the entire keystore under a new Age identity — old key invalidated
- `riversd` must be restarted after any keystore change — there is no hot-reload of secrets in v1

## 11. v2 Upgrade Path

In v2, RPS replaces the local resolver for multi-node deployments. The upgrade requires:

1. Import existing keystore into RPS: `rivers rps import-keystore --from /etc/rivers/riversd.rkeystore`
2. Update `riversd.conf` to point at RPS instead of local file:

```toml
# v1 config
[lockbox]
path       = "/etc/rivers/riversd.rkeystore"
key_source = "env"
key_env_var = "RIVERS_LOCKBOX_KEY"

# v2 config — replace [lockbox] with [rps]
[rps]
endpoint = "https://rps.internal:7443"
trust_bundle = "/etc/rivers/trust-bundle.age"
```

3. Remove `RIVERS_LOCKBOX_KEY` env var from deployment

No handler code changes. No datasource config changes. All `lockbox://` references in `riversd.conf` are valid in both v1 and v2 — RPS resolves them the same way the local resolver does.

Single-node deployments may continue using the local keystore indefinitely — RPS is not required for single-node operation in v2.
