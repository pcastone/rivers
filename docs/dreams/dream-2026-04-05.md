# Dream -- 2026-04-05 (Focus: Production Readiness & Developer Experience)

## Summary

Rivers has matured significantly since v0.50 -- the deploy pipeline (`cargo deploy`), health check system (`riversctl doctor --fix`), per-app logging with rotation, daemon management, and a 55-test regression suite all landed in the last two weeks. The core framework works: you can define TOML configs, point at datasources, and get working REST endpoints without writing code. However, the project sits at a specific inflection point: the **internal development experience is strong** (deep specs, structured CLAUDE.md, disciplined commit history), but the **external experience has gaps** -- a new admin will encounter compilation errors in the test suite, stale branch accumulation, and no `riversctl init` command for bootstrapping a new bundle from scratch. The biggest risk to production readiness is not missing features but missing integration: the pieces exist but several seams are rough (per-app logging just landed, the test suite has compilation errors on `test-coverage` branch, and 47+ stale feature branches create maintenance debt).

### What is Working

- **Deploy pipeline is complete**: `cargo deploy <path>` creates a full runtime directory with binaries, config, TLS certs, lockbox, log directories, and a VERSION file. Both static and dynamic modes work.
- **Doctor system is comprehensive**: 10 health checks covering binary, config, validation, storage, lockbox (with auto-init), TLS (with auto-gen), log dirs, engine/plugin dirs, app log dir, and bundle path. `--fix` repairs 5 of them automatically.
- **Daemon management works**: `riversctl start` prints startup info (binary, config, port, RIVERS_HOME), daemonizes riversd, reports PID. `--foreground` flag for systemd/debug.
- **Per-app logging is production-grade**: AppLogRouter with BufWriter buffering, 10MB rotation, hot-reload flush safety, graceful shutdown flush, and Drop guard. Both V8 and WASM engines write to per-app log files.
- **Config uses absolute paths**: `cargo deploy` canonicalizes the deploy path and writes all config references as absolute paths. Config discovery works from binary location.
- **Packaging infrastructure exists**: systemd unit file, deb/rpm scripts, Justfile with dist-zip, dist-source, release (with symlink + pruning).
- **Documentation is extensive**: 28 spec documents in `docs/arch/`, 21 tutorials in `docs/guide/tutorials/`, installation/quickstart/admin/developer/CLI guides, 3 AI-specific guides, and a feature inventory.
- **Test infrastructure is solid**: Driver conformance matrix (19 tests), V8 bridge contract tests (21 tests), boot parity tests (4 tests), security header tests (3 tests), config validation tests (8 tests).
- **CLAUDE.md is detailed and accurate**: Architecture, crate map, config conventions, bundle structure, spec references, and workflow instructions.

### What Can Be Done Better

- **Branch hygiene**: 47+ local branches, many from completed features (e.g., `fix/ci-sccache-fallback`, `fix/ci-sccache-health`, `fix/ci-sccache-unset` -- three branches for one sccache fix). 30+ remote branches. This creates cognitive overhead and merge confusion.
- **Test suite compilation**: The `test-coverage` branch has compile errors in `rivers-plugin-mongodb` (missing `use` import) and `riversd` (missing `max_rows` field in test struct initializers, missing `init` field in `AppManifest`). Tests that existed before the branch work; the new test files have stale struct references.
- **No `riversctl init` or scaffolding tool**: Creating a new bundle requires manually creating 4+ files with specific TOML structure and conventions. No template or generator exists.
- **Error messages on wrong TOML structure are opaque**: If you use `[views.list_contacts]` instead of `[api.views.list_contacts]`, TOML silently ignores it. No warning is emitted.
- **README has conflicting clone URLs**: Line 53 says `github.com/AquaFlare/rivers.pub.git` while badges point to `github.com/pcastone/rivers`.
- **No structured JSON log output at runtime**: The config has `format = "json"` but the implementation uses `tracing_subscriber` text format. JSON structured logs are needed for production log aggregation.
- **Changelog is sparse**: `todo/changelog.md` only covers the current session's per-app logging work. Historical decisions are spread across dream docs and bug reports.

### What is NOT Working

- **Test suite does not compile on current branch**: `cargo test --workspace` fails with compilation errors in `rivers-plugin-mongodb/tests/lockbox_helper.rs` (missing `ConnectionParams` import) and `riversd` tests (missing `max_rows` and `init` fields in struct initializers). This means CI would fail if `test-coverage` branch were pushed.
- **JS init handlers cannot call `ctx.ddl()`**: The V8 callback for DDL execution is not wired. Documented in `todo/gutter.md` as deferred, but this means init handlers that need to create tables must use a workaround. The WASM engine has the same gap.
- **No graceful shutdown signal handling in `riversctl start --foreground`**: When running as a daemon, there's no PID file and no `riversctl stop` command. The only way to stop riversd is `kill <pid>`.
- **No `riversctl stop` command**: Not implemented. Admins must manually find and kill the process.
- **No Windows CI**: `build-windows.yml` badge exists in README but the file is not in `.github/workflows/`. Only `build-linux.yml`, `build-macos.yml`, and `release.yml` exist.

## Memory Consolidation

### Key Decisions from Recent Work

1. **Per-app logging architecture**: AppLogRouter is a `HashMap<String, BufWriter<File>>` behind a `Mutex`, accessed via `OnceLock<Arc<AppLogRouter>>` global. Apps are keyed by `entry_point` (not `app_name`) because that matches `TASK_APP_NAME` set by the V8/WASM engines. This was a non-obvious alignment requirement discovered during integration.

2. **BufWriter vs per-write flush**: Initially per-write flush was used for correctness, then removed because it defeats the purpose of BufWriter. The solution is explicit `flush_all()` at three points: graceful shutdown, hot reload, and test assertions. This is a pattern to follow for any future buffered I/O.

3. **Engine loader filename mismatch**: The engine loader searched for `librivers_v8` but the crate output is `librivers_engine_v8`. This was a silent failure -- V8 never loaded in dynamic mode, but the system didn't crash because static mode bundles V8 directly. This class of "works in one mode, silently fails in another" is a recurring pattern.

4. **Plugin loader scanning lib/**: The plugin loader was scanning `lib/` and finding engine dylibs, which export a different ABI symbol (`_rivers_engine_abi_version` vs `_rivers_abi_version`). This produced confusing warnings. Fix: plugins only scan `plugins/`, engines only scan `lib/`.

5. **Config absolute paths**: `cargo deploy` now `canonicalize()`s the deploy path and writes all config references as absolute paths. This eliminates the entire class of "works from project root, fails from any other directory" bugs.

### Patterns Established

- **TASK_APP_NAME thread-local**: Set before handler dispatch, cleared via Drop guard. Used by both V8 and WASM engines. This is the mechanism for per-app log routing and should be used for any future per-app context needs (metrics, tracing spans).
- **Doctor check + fix pattern**: Each health check is a function that returns pass/fail/skip, with an optional fix function gated by `--fix`. New checks should follow this pattern.
- **Deploy scaffold pattern**: `cargo deploy` creates the full directory tree, generates config, TLS certs, lockbox, and VERSION file. This is the canonical "what does a Rivers installation look like" reference.

## Pattern Recognition

### Pattern 1: Operational Tooling Follows Features by Weeks

The core Rivers features (DataViews, drivers, V8, auth) were built first. Operational tooling (`cargo deploy`, `riversctl doctor`, per-app logging, daemon management) came weeks later, often triggered by trying to actually deploy and discovering what was missing. This is natural but means the "last mile" of each feature (how does an admin know it's working? how do they debug it?) gets deferred.

**Evidence**: `cargo deploy` was "fully functional" fix in commit `ac3fdca` (April 4), but the engine loader filename mismatch (`dac5516`, April 5) shows that dynamic mode deployment was never actually tested end-to-end before that.

### Pattern 2: Silent Failures Remain the Top Risk

From the previous dream doc: 10 of 14 security findings were silent. This pattern continues:
- Engine loader silently failed to find V8 dylib in dynamic mode
- Plugin loader silently warned about engine ABI but didn't crash
- TOML config silently ignores wrong section prefixes (`[views.*]` vs `[api.views.*]`)
- Per-app logging initially used wrong key (`app_name` vs `entry_point`) -- logs went nowhere, no error

**Heuristic**: Every new subsystem needs a "did it actually work?" confirmation. `riversctl doctor` is the right pattern. Extend it.

### Pattern 3: Test Infrastructure Lags Behind Feature Development

55 new tests were written (good), but the test suite doesn't compile on the current branch (bad). The `max_rows` field was added to `DataViewConfig` but test files that construct this struct weren't updated. This suggests tests are written in isolation and the full `cargo test --workspace` is not run before committing.

### Pattern 4: Documentation is Comprehensive but Spread Thin

There are 28 spec docs, 21 tutorials, 5 guide docs, 3 AI guides, 2 reports, and 2 dream docs. Total: ~60 documents, ~3,000+ lines of guide content alone. The quantity is impressive, but:
- Tutorials reference 14 datasources but only 1 example bundle (address-book with faker)
- The AI guides had 11 stale entries (fixed in a previous pass)
- No single "getting started" path that goes: install -> create bundle -> deploy -> verify

### Pattern 5: Success Builds on Two-Step Workflow

The most productive commits follow a pattern: (1) spec/plan document written first, (2) implementation follows the plan with checkboxes. The test coverage sprint is a clear example -- `tasks.md` has 6 phases with explicit validation steps, and the git history shows systematic phase completion. When this pattern is followed, quality is high.

## Insights

### Production Readiness

**Verdict: 75% ready. The core runtime is solid. The operational shell has gaps.**

What's production-ready:
- `cargo deploy` creates a complete, self-contained directory with absolute paths
- TLS with self-signed cert generation (for dev/staging) and configurable x509 params
- `riversctl doctor --fix` validates and repairs the most common setup issues
- Per-app logging with 10MB rotation and graceful shutdown flush
- systemd unit file with security hardening (NoNewPrivileges, ProtectSystem=strict)
- Connection pooling with circuit breaker across all driver types

What's missing for production:

1. **No `riversctl stop` / PID file**: The daemon writes to `stdout: null` and there's no PID file. Stopping requires `ps aux | grep riversd | kill`. A production service needs `riversctl stop` that reads a PID file, sends SIGTERM, and waits for graceful shutdown.

2. **No structured JSON log output**: Config declares `format = "json"` but the actual output is tracing_subscriber text format. Production log aggregation (ELK, Datadog, CloudWatch) requires structured JSON with timestamp, level, message, and fields.

3. **No log shipping**: Logs go to local files only. No syslog, no journal integration, no OTLP export. The systemd unit sets `LogsDirectory=rivers` which captures stdout, but riversd closes stdout when daemonized.

4. **No health endpoint**: There's no `/health` or `/ready` endpoint for load balancer probes. The admin API has a `/admin/health` but it's commented out in the default config and requires authentication.

5. **No metrics**: No Prometheus endpoint, no StatsD, no OpenTelemetry metrics. Connection pool usage, request latency, cache hit rates, error rates -- all invisible.

6. **TLS cert renewal**: Self-signed certs are generated with configurable `days` but there's no renewal mechanism. Production needs either ACME/Let's Encrypt integration or at minimum a `riversctl tls renew` that regenerates and triggers hot reload.

7. **No config validation on startup with clear error messages**: Config validates in `doctor` but the runtime startup path may not produce equally clear messages. A config error should print the exact line/section that's wrong and exit cleanly.

### New Admin Experience

**Verdict: The deploy-then-doctor path works. Everything else is friction.**

Walk-through of a fresh install:

1. **Get the binary**: `git clone && just build` works. Pre-built releases exist for Linux/macOS. Windows badge exists but CI workflow is missing. An admin on Windows has no pre-built option.

2. **Deploy**: `just deploy /opt/rivers` runs `cargo deploy` which builds everything and creates the directory. This takes 10-20 minutes on first build (compiling V8, all plugins). No progress indication beyond cargo's output. An admin might think it's hung.

3. **First run**: `cd /opt/rivers && ./bin/riversctl doctor` -- this works well. Clear output with PASS/FAIL/SKIP/FIXED for each check. If anything is wrong, `--fix` repairs it. This is the best part of the admin experience.

4. **Start the server**: `./bin/riversctl start` prints startup info and daemonizes. Good. But then: how do you verify it's running? There's no health endpoint. You'd need `curl -k https://localhost:8080/` and hope for a response. If TLS is misconfigured, the error message comes from curl, not Rivers.

5. **Deploy a bundle**: The config has `bundle_path` commented out. The admin needs to: uncomment it, set the path, create the bundle directory, populate it with TOML config files. There's no `riversctl init my-app` or `riversctl bundle create`. The address-book example exists but it's in the source repo, not in the deployed directory.

6. **What trips them up**:
   - The README says `target/release/riversd --bundle address-book-bundle` but a deployed instance has no `target/` directory. The binary is in `bin/` and the bundle goes in `apphome/`.
   - Config file is at `config/riversd.toml` but `RIVERS_HOME` is not set by default. Config discovery works by looking relative to the binary, which is good, but not documented.
   - If they want to use a real database, they need a driver plugin. In static mode, all drivers are built in. In dynamic mode, they need the correct `.dylib` in `plugins/`. The config doesn't tell them which drivers are available.
   - No `riversctl status` command to check if riversd is running, what port, what bundles are loaded.

**Friction points ranked**:
1. No bundle scaffolding tool (highest friction)
2. No `riversctl stop` / `riversctl status`
3. No health endpoint for verification
4. Build time on first deploy (V8 compilation)
5. No clear mapping from "I want to use PostgreSQL" to "what config do I write"

### Developer App-Building Experience

**Verdict: Powerful but requires archaeology to learn. The TOML config is the product, and it needs better on-ramps.**

What works well:
- The address-book example is clean and demonstrates the full pattern: datasource -> DataView -> View -> parameter mapping
- 14 datasource tutorials exist, one per driver type
- The app.toml format is intuitive once you see an example: declare a datasource, declare a DataView that queries it, declare a View that exposes it as an HTTP endpoint
- JSON schemas provide type safety without code

What's painful:

1. **No scaffolding**: `riverpackage` binary exists in the deploy but its purpose isn't clear from the name. It appears to be for packaging, not creating. A developer's first question is "how do I create a new app?" and there's no command for it.

2. **TOML convention traps**: The CLAUDE.md documents several of these:
   - `[api.views.*]` not `[views.*]` (silent failure)
   - `[[data.dataviews.*.parameters]]` array-of-tables with explicit `name` field (not `[data.dataviews.*.parameters.id]`)
   - `ttl_seconds` not `ttl`
   - Datasources use `[[datasources]]` array-of-tables syntax
   
   Each of these is a 30-minute debugging session when you hit it. There's no TOML linter that catches these.

3. **Schema conventions are underdocumented for drivers**: The faker driver uses `"faker"` as the schema attribute key. PostgreSQL uses column types. Redis uses key patterns. The mapping from "what do I put in the schema JSON?" to "what does the driver expect?" requires reading driver-specific tutorial docs.

4. **Error messages from the runtime are often generic**: If a DataView references a datasource that doesn't exist, the error at request time may be "datasource not found" without saying which DataView tried to use it. If a handler module path doesn't resolve, the error may be at startup but buried in log output.

5. **No hot reload for development**: If you change app.toml, you must restart riversd. A `--watch` mode that reloads on file change would dramatically improve the development loop.

6. **CodeComponent handler development is painful**: Writing JS handlers requires knowing the ctx/Rivers API. The V8 bridge tests document what works, but there's no REPL or playground. The AI guide (`rivers-skill.md`) was the primary handler documentation but had 11 stale entries (now fixed).

**What developers hit most (from bugs/ directory)**:
- Parameter binding mismatches across SQL drivers (BUG-004, Issue #54)
- `ctx.dataview()` silently dropping parameters (fixed in PR #48)
- `ctx.app_id` being empty string (fixed)
- Silent TOML config ignoring (ongoing)

### Claude Code Integration

**Verdict: CLAUDE.md is one of the best-structured project guides I've seen. There are specific improvements that would make AI-assisted development even more effective.**

What works well:
- **Architecture section** is accurate and complete: layer descriptions, dual build modes, crate map with roles, bundle structure, config conventions
- **Key Config Conventions** section prevents the silent TOML failures that would otherwise waste AI cycles
- **Spec document index** maps topics to files, reducing search time
- **Workflow section** establishes think-plan-check-execute discipline with todo/changelog tracking
- **The `?` and `??` convention** for manager-mode communication is clever and prevents AI from making unwanted changes

What could be improved:

1. **No "current state" section**: CLAUDE.md describes what Rivers IS but not where it IS in development. Adding a "Current Sprint" section that says "we're on `test-coverage` branch, the test suite has compile errors in X, the next goal is Y" would prevent AI from stumbling into known issues.

2. **No anti-patterns section**: The config conventions tell you what TO do, but not what NOT to do. Adding explicit anti-patterns like "NEVER use `[views.*]` -- it must be `[api.views.*]`" or "NEVER add fields to `HostCallbacks` without bumping `ENGINE_ABI_VERSION`" would prevent high-cost mistakes.

3. **Missing test running instructions**: There's no "how to run tests" section. The workspace has compile errors on the current branch. An AI needs to know: "run `cargo test -p riversd -- --test-threads=1` for integration tests", "skip `rivers-plugin-mongodb` tests until the import is fixed", etc.

4. **The rivers-dev skill references need updating**: The skill exists for building Rivers apps/bundles, but the bugs/docs from the previous dream showed that the skill's examples had 11 stale entries. A periodic audit of skill accuracy should be part of the workflow.

5. **No code example section**: CLAUDE.md references specs but doesn't include small, inline examples of the most common patterns. Adding 3-4 "copy this pattern" examples (new DataView, new View, new handler, new driver test) would reduce the need to search through specs.

6. **Decision log should be in CLAUDE.md, not just changelog.md**: Key architectural decisions (e.g., "entry_point is the canonical app identifier, not app_name", "plugins/ is for driver plugins only, lib/ is for engines only") should be in CLAUDE.md so every session starts with them.

## Art of the Possible

### 1. `riversctl init` -- Bundle Scaffolding Tool
**Impact: High | Effort: Low-Medium**

Add a `riversctl init <bundle-name>` command that creates a complete bundle skeleton:
```
my-bundle/
  manifest.toml          (pre-filled with bundleName, version)
  my-app/
    manifest.toml        (pre-filled with generated UUID, type, port)
    resources.toml       (empty datasources section with comments)
    app.toml             (example DataView + View with comments)
    schemas/
      example.schema.json
```

This eliminates the biggest friction point for both new admins and developers. The address-book-bundle is the template. The code already has all the config structs -- this is just file generation.

Optionally: `riversctl init --driver postgres` pre-fills with PostgreSQL-specific datasource config and schema examples.

### 2. Fix the Test Suite Compilation + CI Gate
**Impact: High | Effort: Low**

Three specific fixes needed:
- Add `use rivers_driver_sdk::ConnectionParams;` to `rivers-plugin-mongodb/tests/lockbox_helper.rs`
- Add `max_rows` field to `DataViewConfig` initializers in `riversd/src/bundle_diff.rs` and `riversd/src/process_pool/tests/context_data.rs`
- Add `init` field to `AppManifest` initializer in `riversd/src/bundle_diff.rs`

Then add a CI step that runs `cargo test --workspace` and blocks merge on failure. This prevents the "tests were green when I wrote them but broke when a struct changed" pattern.

### 3. `riversctl stop` + PID File + `riversctl status`
**Impact: High | Effort: Low**

- Write PID to `<RIVERS_HOME>/run/riversd.pid` on daemon start
- `riversctl stop`: Read PID file, send SIGTERM, wait up to 30s, report
- `riversctl status`: Check PID file, verify process is alive, print port/uptime/bundle info

This completes the admin lifecycle: `doctor` -> `start` -> `status` -> `stop`. The code is straightforward -- `start.rs` already has 80% of the infrastructure.

### 4. TOML Config Linter (`riversctl lint <bundle-path>`)
**Impact: Medium-High | Effort: Medium**

Validate bundle TOML beyond syntax:
- Warn on `[views.*]` (should be `[api.views.*]`)
- Warn on datasource referenced by DataView that doesn't exist in resources.toml
- Warn on handler module path that doesn't resolve to a file
- Warn on init handler with DDL but no `ddl_whitelist` entry
- Warn on duplicate DataView names across apps in a bundle
- Validate JSON schema files referenced by DataViews exist and parse

This catches the entire class of "silent TOML misconfiguration" bugs before startup. The `riverpackage` crate could be extended for this, or it could be a `riversctl lint` subcommand.

### 5. Structured JSON Logging + Health Endpoint
**Impact: Medium | Effort: Medium**

Two changes for production observability:
- Wire `tracing_subscriber::fmt::format::json()` when `format = "json"` in config. Each log line becomes `{"timestamp":"...","level":"INFO","target":"riversd::server","message":"...","app":"my-app","trace_id":"..."}`.
- Add a `/health` endpoint that returns `{"status":"ok","version":"0.52.8","uptime_seconds":3600,"bundles":2,"apps":4}`. No auth required. Configurable via `[base.health]` section.

These are table-stakes for any production deployment behind a load balancer or in a container orchestrator. The structured logging change is ~20 lines in the tracing subscriber setup. The health endpoint is a single Axum handler.
