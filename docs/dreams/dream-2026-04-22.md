## Summary

### What is Working

- **Two of three P0s landed cleanly.** `v8_str_safe` helper replaced all 14 `.unwrap()` sites in `context.rs` — the remaining single `.unwrap_or_else` is inside the helper's own definition and is the fallback path, not a panic site. `TransactionCommitFailed` is a distinct `TaskError` + `ViewError` variant with a `transaction_state = "unknown"` surface that reaches the client regardless of debug mode. Both fixes are the shape the prior dream asked for.
- **Root-cause fixes during compaction.** Two pre-existing bugs were found and fixed properly rather than papered over: (a) `sync_from_app_context` was called *after* init handlers in `bundle_loader/load.rs` — moved *before*, so `ctx.ddl()` now works for init-time DDL; (b) `host_callbacks.rs` was creating/tearing down a `tokio::runtime::Runtime` per dataview call, which was killing mysql_async pool tasks and `tokio_postgres` connection tasks between queries. Removing the per-call runtime fixed a whole class of driver-internals-got-shut-down failures.
- **Broker bridge has a clean architectural split.** `DatasourceToken::Broker` variant, `BROKER_DRIVER_NAMES` gate in the runtime, per-task producer cache in a thread-local mirroring the direct-dispatch pattern. Code shape is consistent with the rest of the direct-dispatch surface — reviewable against a known template.
- **Canary survived major refactors.** Went from 22 PASS → 89 PASS across the session despite landing a new V8 callback surface, a new token type, a new scenario harness, and two deep bug fixes. Regression shielding actually worked.
- **P0 fix had its own tests.** `p0_2_transaction_commit_failed_exposes_unknown_state` + `p0_2_commit_failed_not_sanitized_in_release` lock in the commit-failure behaviour. Not comprehensive (see below) but not absent.

### What Can Be Done Better

- **Broker publish hot path double-encodes JSON.** `rivers_broker_publish_callback` does `v8::json::stringify` → `to_rust_string_lossy` → `serde_json::from_str` on every publish, then again on extract payload. For an order-event publish that's two UTF-8 re-scans + two JSON-parse passes per message. Direct v8-to-serde via `v8::json::stringify` writing into a `Vec<u8>` buffer is possible; `deno_core` does it. For a hedge-fund event stream at 10k msg/s this is the tail-latency bomb.
- **`.expect("producer initialised above")` at broker_dispatch.rs:129.** Locally justified by the immediately-preceding `borrow_mut` insert, but it's still an `.expect` at the V8 FFI boundary. If the invariant ever breaks (refactor, unexpected re-entry), it becomes the same SIGABRT class the prior dream flagged. `ok_or_else` → `throw_error` is one line and has the same outcome without the panic vector.
- **Direct `mysql_async::Conn::new` per dataview call eliminates connection pooling entirely.** The Pool change was motivated by runtime-isolation-induced task-teardown. The runtime-isolation was the actual bug, and it's now fixed (host_callbacks.rs). The Pool swap addressed the *symptom*, not the cause — and left MySQL with a full TCP + auth handshake per dataview call. For a hedge-fund workload where even the research-data side does thousands of queries/sec, this is unacceptable. The Pool change should be reverted now that the runtime fix is in.
- **Broker-bridge startup blocks on `create_consumer().await`.** `wire_streaming_and_events` awaits Kafka's initial connect inline during bundle load. If the broker is unreachable (today, on macOS: "No route to host" with Kafka cluster 192.168.2.203 reachable via `nc`), riversd hangs before HTTP listener ever binds. A canary node's Kafka being flaky blocks PG, MySQL, filesystem, and HTTP canaries from running.
- **Commit-failure thread-local is one-shot.** `TASK_COMMIT_FAILED` is a `Cell<Option<(String, String)>>`. If a handler calls `ctx.transaction()` twice in one request and both commits fail, the second overwrites the first silently. Unlikely today, but the spec doesn't forbid it.
- **`sync_from_app_context` ordering fix was an artefact of debugging, not a policy.** The init-handler loop could regress if someone moves code around in `load.rs`. A doc comment saying "must be called before init_handler_loop" is not a compile-time guarantee. An ordering-invariant test, or refactoring so `ctx.ddl()` validates its own prerequisite at call time, would be the durable fix.
- **JSON stringify of error payloads uses `to_rust_string_lossy`.** Lossy UTF-8 in error messages means a byte that doesn't round-trip gets replaced silently. For forensics on failed trades, lossy is a bug. Prefer `to_rust_string` + explicit error on invalid UTF-8 — or guarantee the upstream payload is ASCII.

### What is NOT Working (project-review blockers)

- **P0-3 supervisor is half-done.** Commit `0435efc` added `catch_unwind` around swc compile but **explicitly deferred the timeout** ("Follow-up recommended but not blocking ... spawn_blocking + tokio::time::timeout"). The prior dream's P0 was specifically *both*: panic-containment AND timeout-bound. swc panics are one failure class; swc hangs / pathological complexity are another. A crafted handler that generates a 50MB TS file with deeply nested conditional types will spin swc forever — `catch_unwind` doesn't stop it. At bundle load that blocks the deploy pipeline; at hot-reload that wedges a worker. **Re-open P0-3 as `P0-3b — compile-time bound`.**
- **Broker consumer startup is a global-availability blocker.** `wire_streaming_and_events` calls `create_consumer(params, cfg).await` inline. If the broker is unreachable, bundle load hangs indefinitely — and with it, **every other app in the bundle**. In a fund running a multi-app deploy (orders, market-data, risk, audit) where one app uses Kafka, a Kafka flake takes out the whole node. This was directly observed in the current session: canary hangs on startup because rskafka reports "No route to host" despite TCP reachability. **New P0.** Broker consumer init must run in a spawned task that retries; startup must not depend on it.
- **MySQL lost its connection pool.** With `mysql_async::Conn::new` per call, each dataview pays a full handshake. The change was load-bearing for the session (fixed the `Tokio 1.x context shutting down` error), but now that host_callbacks no longer tears down runtimes, the Pool should come back. **New P0 for finance prod.** Every low-latency workload needs pooled connections with a hot-spare minimum.
- **Broker producer lazy-init inside V8 callback blocks a worker thread on network I/O.** First `publish()` in a task triggers `rt.block_on(driver.create_producer(...).await)` on a V8 worker. Producer creation involves broker metadata lookup — tens to hundreds of milliseconds on Kafka's initial round-trip. A V8 worker pool is typically small (1–4 per CPU); blocking one on network I/O serialises the whole pool against a single broker round-trip. **P0.** Producer pre-warming at bundle load + hand-off to the task via thread-local is the right pattern, not lazy-init-on-first-use inside a callback.
- **Topic-at-create-time coupling in kafka plugin.** During this session, making scenarios work required setting `canary-kafka.database = "kafka_consume"` because the kafka plugin binds producer topic from `params.database` at create-time. The `OutboundMessage` struct carries a `destination` field for exactly this purpose; the driver should use it. Today the plugin violates the contract silently — other brokers (rabbitmq, nats) may or may not, nobody knows because the canary stumbled into the one that doesn't. **P1 for spec correctness, P0 for a fund running multiple topics from one datasource** (which is the standard pattern).
- **Thread-local registries still lack a panic-safety test.** Prior dream flagged this. `TASK_DIRECT_BROKER_PRODUCERS` + `TASK_DIRECT_DATASOURCES` + `TASK_TRANSACTION` + `TASK_MODULE_NAMESPACE` + `TASK_MODULE_REGISTRY` + `TASK_COMMIT_FAILED` — six thread-locals, each cleaned up in `TaskLocals::drop`. No test panics mid-task and asserts the clean-up path. In Rust `Drop` runs even on panic, but *order* matters (the broker producer must `close().await` before the RT handle is dropped). If any future `Drop` re-orders implicitly, it will leak until the worker is recycled.
- **No property-test corpus for swc still.** Prior dream flagged. Commit `0435efc` acknowledged the follow-up and deferred it. Every day that passes is another day a handler author (or a CI-injected compile) could feed swc something novel.

## Pattern Recognition

### Codex-characteristic pathologies observed in round 2

**Pattern 1: Fix the symptom, not the cause.**
The MySQL `Pool::new` → `Conn::new` swap is the cleanest example. The actual bug was in `host_callbacks.rs` (per-call Runtime tearing down driver tasks). Working the fix from the driver end made MySQL work *and* broke connection pooling *and* left PostgreSQL fragile for the same reason (PG was "fixed" by removing the runtime isolation; MySQL was "fixed" by changing the driver). Two fixes for one bug, with different blast radii. A reviewer must ask: "for each symptomatic fix, what was the underlying cause, and is every manifestation of that cause now fixed?"

**Pattern 2: Incremental inline fixes replace atomic commits.**
When a long session accrues many small fixes to get a system running, the resulting history is un-bisectable and the intermediate states are un-reviewable. Ideal: each fix is its own commit with test + message. Observed: a single compacted session's worth of edits with no commit trail. For a regulated environment this is a **process** problem, not a code problem — and no amount of code review will fix it.

**Pattern 3: Runtime validation as test substitute.**
Canary passing is evidence the code *can* work, not that it *will always* work. The broker publish path has zero integration tests; confidence rests on "the canary stream app publishes and consumes." If the canary stops running (the infra hangs again), the signal vanishes. Tests that run on every CI pass are the durable artefact.

**Pattern 4: Comments protecting invariants instead of code.**
`host_callbacks.rs` now says "cdylib plugins are disabled in this build — comment for why the previous spawn_blocking+Runtime::new." True today. In 6 months when Plugin ABI v2 lands, someone re-enabling cdylib plugins without restoring the isolation will regress. A compile-time guard (`#[cfg(feature = "static-engines")]` gate or similar) would enforce the invariant; a comment documents it.

**Pattern 5: `block_on` inside hot-path callbacks.**
`broker_dispatch.rs` uses `rt.block_on(producer.publish(message))` on every publish. This is correct by construction (V8 callbacks are synchronous; you must block to await), but every such block blocks a V8 worker on network I/O. In the hedge-fund context this must be audited per site: producer publish, dataview execute, ddl execute, direct-dispatch filesystem. Each one is a potential tail-latency outlier.

### Success patterns

- **Thread-local cleanup in `Drop` for the broker producer** (`prod.close().await` before dropping). The compaction summary records the user explicitly made this ordering correct ("closed+cleared in Drop via RT_HANDLE before clearing"). That is a load-bearing invariant; getting it right without a test is lucky — but the code shape is right.
- **Per-topic EventBus publish in `broker_bridge.rs`.** Adding `topic_event = Event::new(&msg.destination, payload)` alongside the generic `BROKER_MESSAGE_RECEIVED` event means MessageConsumer subscriptions match by topic name. This is the shape the spec asked for.
- **DDL whitelist three-gate remained intact.** Even with the init-handler timing bug fix, the three-gate check (DDL whitelist, init-only, app_id matching) wasn't weakened. Security boundary held through a major refactor.

## Insights

### Review heuristics for round 2

**H1. Ratio of follow-up actions closed / identified.**
Dream 2026-04-21 identified 3 P0s and 7 P1s. Round 2 closed 2.5 P0s, 0 P1s. For a production-critical codebase, the rate of P1-closure matters as much as P0-closure: P1s are the indicators that a team has durable review discipline. This one doesn't.

**H2. Fraction of new code covered by tests.**
~1000 lines new since the prior dream. ~150 lines of new tests (token variant, `build_outbound_message`). **~15% test coverage on new code.** For a finance workload I'd want 80%+ including integration tests that don't require external infra (use a mock broker or testcontainers-rs).

**H3. Number of new panic/unwrap sites on the hot path.**
- `broker_dispatch.rs`: 1 `.expect` at line 129 (hot path)
- `host_callbacks.rs` post-change: need to re-audit. Previously the per-call Runtime was a defence against cdylib panics escaping into the host runtime; removing it widened the blast radius of any future plugin panic. At least re-audit before Plugin ABI v2 ships.

**H4. Number of "foot-gun" thread-local ordering invariants.**
Was 4, now 6. Each new thread-local is a new ordering invariant. The broker producer must close before its RT handle drops. No test proves this holds under panic.

**H5. Latency-budget audit of new hot-paths.**
`broker_dispatch.rs` publish path:
- `v8::json::stringify` (allocates, serialises V8 tree to UTF-16 string): ~50 µs for a 1KB message
- `to_rust_string_lossy` (re-encodes to UTF-8): ~20 µs
- `serde_json::from_str` (parses to serde_json::Value): ~100 µs
- `build_outbound_message` (field extract + another serialise of payload): ~100 µs
- `rt.block_on(producer.publish(message))` (Kafka ack): 1–50 ms
- Return `rv.set(json_to_v8(...))`: ~50 µs

Hot-path overhead excluding broker: ~320 µs. The double-JSON round-trip is ~200 µs of that. A fund running 10k publishes/sec pays **2 seconds of CPU per second** just on JSON format churn. P99 won't survive that.

### Process improvements

- **No more compacted-session mega-patches.** When a debug session produces >5 fixes, each should land as its own commit with test+message before the session ends, or be PR-captured for a later atomic pass. The current branch has ~1000 lines of fixes that will be impossible to bisect if a regression is found.
- **"Symptom vs. cause" audit prompt.** For every bug fix, the reviewer should ask aloud: "what was the actual cause? does every manifestation of that cause have a fix? is there a test that would catch the cause regressing?" The MySQL Pool change would not have landed under that prompt.
- **Compile-gate invariants for removed defences.** Removing the per-call `Runtime::new` isolation left a `#[cfg(feature = "static-engines")]` assumption as a comment. Promote it to a compile-time assertion (`#[cfg(not(feature = "static-engines"))] compile_error!("Plugin ABI v2 requires restoring runtime isolation")`).
- **Latency-budget spec per hot path.** Publish, dataview, ddl, direct-dispatch each get a documented P99.9 budget. Every change to those paths is reviewed against the budget. Today there's no number to violate.

## Art of the Possible

Carrying the prior dream's list forward, adding two round-2-specific capabilities.

### 1. Broker pre-warm + non-blocking consumer startup (P0 fix reframed as capability)

**Problem:** Any broker flake hangs bundle load. Publish pays network I/O in V8 callback.
**Capability:** `wire_streaming_and_events` spawns consumer tasks without awaiting initial connect; the task handles retry internally. Producers are created during bundle load (one per broker datasource) and stashed in a shared registry; task start-up hands off the pre-warmed producer via thread-local.
**Builds on:** existing `BrokerConsumerBridge::run` loop (already handles reconnect internally; just needs startup moved into it).
**Complexity:** 1 day. Value: **unblocks the current session's stuck canary** and generalises to every broker-consuming app.

### 2. Integration-test harness for V8 callback paths (P1 infrastructure)

**Problem:** ~1000 lines of new V8 callback code have ~150 lines of tests, and the tests that exist cover only the pure sub-functions. The callback surface itself is only tested via canary-against-real-Kafka.
**Capability:** A testing pattern that sets up a bare V8 isolate, registers the `Rivers.__brokerPublish` callback against a mock `DriverFactory` with a mock `MessageBrokerDriver`, and asserts the envelope shape, error propagation, and thread-local cleanup. This closes the coverage gap without requiring broker infra.
**Builds on:** existing unit-test patterns in `broker_dispatch.rs` (already tests `build_outbound_message`; extend to the callback).
**Complexity:** 1 day. Value: **closes the round-2 coverage gap** and makes every future V8 callback surface easier to review.
