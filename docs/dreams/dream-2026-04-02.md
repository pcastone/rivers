# Dream — 2026-04-02 (Focus: Bugs Since v0.50)

## Summary

Since v0.50.2.2 (March 23) through v0.52.7 (April 2), Rivers has undergone 121 commits across 53 PRs. The bug landscape reveals three distinct categories: **security design gaps** (systemic, found via audit), **integration mismatches** (spec vs implementation divergence), and **infrastructure friction** (CI/tooling). The most concerning pattern is that several bugs were invisible — they didn't crash, they silently did the wrong thing.

### What is Working
- **Driver architecture** — the trait-based Connection/DatabaseDriver design made it possible to apply DDL guards across 16 drivers in a single pass
- **Layered isolation** — V8 sandbox, parameterized queries, typed QueryValue enum all prevent classic injection
- **Test infrastructure** — 17 live driver tests against the podman cluster catch real integration issues
- **Security audit process** — the structured audit found 17 issues, 14 now resolved

### What Can Be Done Better
- **Parameter binding convention** — the single biggest design gap. Every SQL driver handles `$name` differently, and spec examples don't match any driver's reality
- **Spec-to-code validation** — too many spec examples describe behavior that doesn't exist (ctx.session, Rivers.http, async handlers, streamDataview)
- **CI resilience** — three PRs (#39, #40, #41) were needed to make sccache not crash builds

### What is NOT Working
- **Parameter binding across SQL drivers** — `$name` placeholders from spec examples fail on every SQL driver. PostgreSQL and MySQL silently bind in wrong order. SQLite fails with prefix mismatch. Issue #54 filed.
- **ctx.dataview() was silently dropping parameters** — handlers calling `ctx.dataview("name", {id: 42})` had params ignored. Fixed in PR #48, but this was undetected for the entire 0.50.x lifecycle.

## Pattern Recognition

### Pattern 1: Silent Failures are the Norm

10 of 14 security findings and 4 of 11 integration bugs were **silent**. The system didn't crash, didn't log, didn't warn. It just did the wrong thing. This is the most dangerous pattern because:
- Users don't report what they can't see
- Tests pass because they don't test for absence of guardrails
- Code review misses "what's NOT there"

**Heuristic:** For every new feature, ask "what happens if someone misuses this?" and "does the system tell them?"

### Pattern 2: Spec Drift Creates Ghost APIs

Six documented APIs didn't exist (ctx.session, streamDataview, Rivers.http, async handlers, hot_reload config, OTel config). These accumulated over time as specs were written ahead of implementation.

**Heuristic:** Every spec example should have a corresponding test. If the test doesn't exist, the feature doesn't exist — remove it from docs.

### Pattern 3: Parameter Binding is a Cross-Cutting Concern

The SQLite bug report revealed that ALL SQL drivers have parameter binding issues — but each in a different way. The root cause is that parameter translation was left to each driver instead of being centralized.

**Heuristic:** When you find a bug in one driver, check all drivers for the same class of issue.

### Pattern 4: CI Takes 3x the Expected Effort

The sccache fix took 3 PRs. The Windows fix required an issue + test ignores + workflow removal. The gitignore fix required emergency intervention. CI problems are never "one quick fix."

**Heuristic:** Budget 3x for CI changes. Always add fallback/graceful-degradation.

## Insights

### 1. The V8 Sandbox Was More Open Than Anyone Realized

The V8 engine had no timeout, no heap callback, available code generation from strings, and a non-constant-time comparison function. It was a sandbox with the door unlocked. The fix was straightforward (4 changes), but the gap existed since the engine was first written.

**Lesson:** Sandboxes need negative testing — test what SHOULDN'T work, not just what should.

### 2. The Parameter Binding Problem is Architectural

Every SQL driver invented its own parameter handling. PostgreSQL sorts alphabetically. MySQL does the same. SQLite auto-prefixes. Cassandra had the same alphabetical bug and fixed it independently. This should be solved once in the DataView engine, not repeatedly in each driver.

**Lesson:** When three drivers solve the same problem three different ways, the abstraction is in the wrong place.

### 3. Documentation Was the Biggest Source of "Bugs"

11 items in the doc audit were wrong or described non-existent features. This isn't a code bug — it's a process bug. Docs were written as aspirational, not descriptive.

**Lesson:** Docs should be generated from or validated against code. At minimum, every code example in docs should be a test.

### 4. The Security Audit Pattern Works

The structured audit (read code → check OWASP/CIS/NIST → report findings → fix by severity) found 17 issues in one pass. 14 were fixed within the same session. This is repeatable and should be done quarterly.

## Art of the Possible

### 1. Automated Spec-Code Validation
Build a tool that extracts code examples from spec/doc markdown files and runs them as tests. If `rivers-skill.md` shows `ctx.dataview("name", {id: 42})`, there should be a test proving this works. Complexity: Medium. Value: High — prevents ghost APIs.

### 2. Parameter Translation Engine (Issue #54)
Centralize `$name` → native format translation in the DataView engine. Each driver declares `param_style()`, the engine rewrites. This fixes all SQL drivers at once and makes the spec examples actually work. Complexity: Medium. Value: Critical — blocks real usage.

### 3. Security Regression Test Suite
Convert all 17 audit findings into automated tests: "DDL rejected on execute()", "infinite loop terminates in <6s", "error response doesn't contain 'postgres'", etc. Run on every PR. Complexity: Low. Value: High — prevents regression.

### 4. Bundle Linter
A `riverpackage lint` command that validates not just TOML syntax but semantic correctness: "this DataView references a datasource that doesn't exist", "this handler module path doesn't resolve", "this init handler has DDL but no whitelist entry". Complexity: Medium. Value: Medium.

### 5. Rivers Playground (Docker Compose)
A single `docker-compose up` that runs riversd + all test services + a sample bundle. New contributors can see Rivers working in 60 seconds. The podman cluster at 192.168.2.161 proves the infrastructure works — just needs packaging. Complexity: Low. Value: High for adoption.
