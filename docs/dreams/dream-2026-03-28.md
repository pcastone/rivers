# Dream - 2026-03-28 (Focus: Patterns of Concern)

**Project:** Rivers v0.52.5
**Scope:** Full implementation review — architectural health, code patterns, security posture, growth trajectories

---

## Summary

Rivers is architecturally sound and test-rich (1,218 tests, zero unsafe blocks, zero production unwraps). The recent keystore and ExecDriver features demonstrate healthy incremental delivery. However, several growth patterns are emerging that will compound if not addressed before the Wave 0-6 remediation plan completes.

### What is Working

- **Zero unsafe, zero panic-unwrap discipline** — every error path is handled idiomatically. This is rare in a 23K-line server binary.
- **Plugin isolation** — new features (keystore, exec) land as separate crates with independent test suites. No viral changes across the codebase.
- **Spec-driven development** — 23 architectural specs, deviation tracking, gap analysis before feature completion. The spec-to-code traceability is unusually strong.
- **Security-first encryption** — keystore key bytes never cross to V8/WASM, nonces are OsRng-only, auth tag failures are generic. The ExecDriver has SHA-256 pinning, privilege drop, no-shell execution, and bounded I/O. These aren't afterthoughts.

### What Can Be Done Better

- **AppContext is a god object** (24 fields, growing with each feature). It should be decomposed before Wave 5 adds more.
- **Dispatch wiring gap pattern** — both keystore and exec driver required post-hoc fixes to wire context through the handler dispatch path. The `TaskContextBuilder` has 15 setter methods but dispatch sites only call 3. New capabilities need an enrichment layer, not manual wiring.
- **Test coverage gaps in older crates** — `rivers-storage-backends` and `rivers-lockbox-engine` have zero tests. These are security-critical components.
- **Large files approaching split thresholds** — `v8_engine.rs` (2.1K), `server.rs` (2.4K), `engine_tests.rs` (3.3K). None are painful yet, but all are on a growth trajectory.

### What is NOT Working (Landmines)

- **DataView cache is built but never wired** — `TieredDataViewCache` exists, types exist, but the executor never uses them. This is a performance landmine under load.
- **SSE and WebSocket are stubs** — SSE sends hardcoded "heartbeat" instead of driving a CodeComponent handler. WebSocket has infrastructure but no actual frame loop. These block Wave 4 and will surprise anyone trying to use them.
- **Polling on_change handler is configured but never called** — config parsing works, but the dispatch path is a no-op. Silent failure.
- **Error response envelope inconsistency** — some paths return `{code, message}`, others return `{"error":{...}}`. This will break client contracts.

## Pattern Recognition

### Pattern 1: Dispatch Wiring Gap (Recurring)

**Symptom:** New capabilities are added to `TaskContext` and `TaskContextBuilder`, but the 13+ dispatch sites in `view_engine.rs`, `guard.rs`, `graphql.rs`, etc. never call the new builder methods.

**Occurrences:**
- Keystore: `ctx.keystore` was always `None` until a shared resolver fallback was added
- Lockbox: `.lockbox()` builder method exists but is never called from any dispatch site
- Storage: `.storage()` builder method exists but is never called
- Driver factory: `.driver_factory()` exists but only called in some paths

**Root cause:** The builder pattern requires explicit opt-in at every dispatch site. There's no "enrich with all available capabilities" step.

**Fix pattern:** Create an `enrich_task_context()` function that takes `AppContext` + a builder and automatically wires all available capabilities. Call it from a single point instead of 13+ sites.

### Pattern 2: Config-Exists-But-Never-Called

**Symptom:** Configuration types parse correctly, validation passes, but the runtime never reads the configured values.

**Occurrences:**
- Polling `on_change` handler: config parsed, never dispatched
- SSE push handler: config parsed, hardcoded heartbeat sent instead
- DataView cache: `TieredDataViewCache` built, never used by executor
- CSRF manager: config parsed, manager built, never checked in middleware

**Root cause:** Bottom-up implementation (parse config first, wire runtime later) leaves gaps when the "wire later" step is deferred.

### Pattern 3: Growing God Objects

**Symptom:** Central structs accumulate fields with each feature, becoming harder to reason about.

**Occurrences:**
- `AppContext`: 24 fields, 3 added for keystore, more coming for sessions/CSRF
- `TaskContext`: 14 fields (gated by features), growing with each new capability
- `SerializedTaskContext`: 16 fields, must stay in sync with TaskContext

**Trajectory:** By Wave 5, `AppContext` will have ~30 fields. Not blocking yet, but extraction should be planned.

### Pattern 4: Test Coverage Inversely Correlates with Age

Newer crates have excellent coverage. Older core crates have gaps:
- `rivers-keystore-engine` (new): 60 tests
- `rivers-plugin-exec` (new): 107 tests
- `rivers-storage-backends` (old): 0 tests
- `rivers-lockbox-engine` (old): 0 tests

The storage backends are security-critical (session storage, cache backend). Zero tests is a landmine.

## Insights

### Lessons Learned

1. **Gap analysis catches real issues** — both features required post-implementation fixes found by spec-vs-code gap analysis. The keystore dispatch wiring gap would have been a production bug.

2. **Parallel subagent development works** — independent crate modules (integrity, template, schema, executor) were built in parallel without conflicts. The boundaries were clean because each module had one responsibility.

3. **The driver plugin pattern scales** — ExecDriver was the 11th plugin crate and followed the exact same pattern (Cargo.toml, DatabaseDriver trait, Connection trait, ABI exports). No framework changes needed.

4. **Spec docs need updating alongside code** — the ExecDriver spec had 3 errors found during gap analysis (typo, missing field on QueryResult, wrong CLI signature). Specs rot if not maintained.

### Heuristics for Future Work

- **Always gap-analyze before marking complete** — the dispatch wiring gap was invisible until the gap analysis ran
- **Test the dispatch path, not just the module** — unit tests on encrypt/decrypt passed, but the JS handler couldn't call them because the context wiring was missing
- **New capabilities need a dispatch-site audit** — any time a `TaskContext` field is added, check all 13+ builder call sites
- **Split files at 2.5K lines** — v8_engine.rs and server.rs are approaching this threshold

## Art of the Possible

### Idea 1: TaskContext Auto-Enrichment Layer

**What:** A single `fn enrich(builder: TaskContextBuilder, ctx: &AppContext, app_id: &str) -> TaskContextBuilder` that automatically wires all available capabilities (keystore, lockbox, storage, driver_factory, dataview_executor) based on what's configured.

**Builds on:** Existing `TaskContextBuilder` pattern, `AppContext` struct
**Complexity:** Low (1 day)
**Value:** High — eliminates the recurring dispatch wiring gap pattern forever

### Idea 2: Plugin Health Dashboard

**What:** An admin endpoint (`/admin/plugins`) that shows registered drivers, their connection status, integrity check results (for ExecDriver), and keystore status.

**Builds on:** Existing admin API, DriverFactory, KeystoreResolver, ExecDriver CommandIntegrity
**Complexity:** Medium (3 days)
**Value:** Medium — operational visibility into the growing plugin ecosystem

### Idea 3: DataView Result Streaming

**What:** Instead of buffering entire QueryResult in memory, stream rows as NDJSON. The DataView executor already has a `stream_execute` path — wire it to SSE views.

**Builds on:** Existing DataViewExecutor, SSE infrastructure, streaming REST views
**Complexity:** High (5 days, part of Wave 4)
**Value:** High — enables large result sets without memory pressure

### Idea 4: Keystore Key Rotation Webhook

**What:** When a key is rotated via `rivers-keystore rotate`, emit an EventBus event that handlers can subscribe to. Enables automated re-encryption workflows.

**Builds on:** Existing EventBus, keystore resolver, message consumer views
**Complexity:** Medium (2 days)
**Value:** Medium — automates the lazy re-encryption pattern from the keystore spec

### Idea 5: ExecDriver Script Sandbox via Landlock (Linux)

**What:** Use Linux Landlock LSM to restrict ExecDriver script filesystem access beyond just `run_as_user` permissions. Declare allowed paths in config.

**Builds on:** Existing ExecDriver process isolation, `pre_exec` hook
**Complexity:** Medium (3 days, Linux-only)
**Value:** High for security-sensitive deployments — defense in depth beyond OS permissions

---

## Closing Thought

Rivers has strong bones. The zero-unsafe, zero-unwrap discipline and spec-driven development process are paying off — new features land cleanly without destabilizing existing code. The main risk isn't a single catastrophic flaw; it's the accumulation of "config parsed but never wired" gaps and the growing AppContext. The Wave 0-6 remediation plan addresses most of this, but the dispatch wiring pattern needs a structural fix (auto-enrichment) before more capabilities are added. The codebase is ready for the next phase — it just needs the plumbing tightened before the pressure increases.
