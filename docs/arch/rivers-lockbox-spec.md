## 1. Overview

LockBox is Rivers' secret management subsystem. In v1, secrets live in a local Age-encrypted keystore file managed by the `rivers lockbox` CLI and decrypted by `riversd` at startup.

The keystore file is a first-class operational artifact — versioned, backed up, and managed via CLI. It is not an ad-hoc `.env` file.

Datasources reference secrets via `lockbox://` URIs in `riversd.conf`. At startup, `riversd` loads the keystore entry index (name/alias to entry position) into host memory — secret values are never held in memory. When a datasource connection is established, the host resolves the `lockbox://` URI by reading the entry from disk, decrypting it on demand, using the value, and immediately zeroizing it. The raw credential never enters the ProcessPool isolate — it is held host-side transiently and resolved opaquely. <!-- SHAPE-5 amendment: no in-memory value caching -->

```
riversd.conf
  credentials_source = "lockbox://postgres/orders-prod"
          │
          ▼
     riversd startup
          │
          ├─ read .rkeystore from configured path
          ├─ decrypt using key source (env var / key file / agent)
          └─ load name/alias → entry index into host memory (no secret values)
                    │
                    ▼
            datasource init
                    │
                    ├─ resolve "postgres/orders-prod" → entry index
                    ├─ fetch_secret_value(index) → decrypt from disk, use, zeroize
                    ├─ establish connection pool using credential
                    └─ decrypted value zeroized immediately after use
                              │
                              ▼
                      opaque token issued to ProcessPool
```

## 2. Keystore File Format

### 2.1 Physical format

The keystore is a **TOML document encrypted with Age**. The plaintext TOML is never written to disk — the encrypted file is the only on-disk representation.

File extension: `.rkeystore`

```
riversd.rkeystore
```

The encrypted payload is a TOML document. Age handles encryption, authentication, and key management (X25519 + ChaCha20-Poly1305 AEAD). Rivers does not implement its own crypto envelope.

```bash
$ file riversd.rkeystore
riversd.rkeystore: Age encrypted file
```

### 2.2 Plaintext TOML schema

```toml
# Rivers Keystore — plaintext inside Age envelope
# Do not edit this file directly. Use: rivers lockbox <command>

version = 1

[[entries]]
name    = "postgres/orders-prod"
value   = "postgresql://user:password@host:5432/orders"
type    = "string"
aliases = ["db/orders", "orders-db"]
created = "2026-01-15T10:00:00Z"
updated = "2026-01-15T10:00:00Z"

[[entries]]
name    = "anthropic/api_key"
value   = "sk-ant-..."
type    = "string"
aliases = ["llm_key", "anthropic_key"]
created = "2026-01-15T10:00:00Z"
updated = "2026-01-15T10:00:00Z"

[[entries]]
name    = "jwt_signing_key"
value   = "base64url:..."
type    = "base64url"
aliases = ["jwt_key", "signing_key"]
created = "2026-01-15T10:00:00Z"
updated = "2026-01-15T10:00:00Z"
```

## 3. Entry Model

### 3.1 Fields

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | string | yes | Canonical name. Unique within keystore. Used in `lockbox://name` references. |
| `value` | string | yes | The secret value. Plaintext string inside the encrypted envelope. |
| `type` | string | yes | Value type hint for host-side handling (see §3.2). |
| `aliases` | string[] | no | Alternative names that resolve to this entry. May be empty array. |
| `created` | ISO 8601 | yes | Set by CLI at creation. Never modified. |
| `updated` | ISO 8601 | yes | Set by CLI on creation and value change (`rivers lockbox rotate`). |
| `driver` | string | no | Driver name (e.g., `"postgres"`, `"redis"`, `"kafka"`). |
| `username` | string | no | Database username or bind DN. |
| `hosts` | string[] | no | Host list as `"host:port"` strings. Supports clusters with multiple entries. |
| `database` | string | no | Database, bucket, or base DN name. |

Credential record fields (`driver`, `username`, `hosts`, `database`) are optional. Existing password-only keystores remain valid. When present, these fields enable full datasource connection resolution from the keystore — bundles can move between environments by swapping the LockBox with no hardcoded IPs or usernames in config.

### 3.2 Value types

| Type | Description | Delivery to driver |
|---|---|---|
| `string` | Arbitrary string — API keys, passwords, tokens, connection strings | Raw string |
| `base64url` | Base64url-encoded binary — signing keys, symmetric keys | Decoded bytes |
| `pem` | PEM-encoded certificate or private key | Raw PEM string |
| `json` | JSON object — structured credentials (OAuth client secrets, service account keys) | Parsed object |

Type determines how the host delivers the resolved value to the datasource driver. The CodeComponent isolate never receives the raw value regardless of type.

### 3.3 Meta Sidecar Files

For test and development environments, non-secret connection metadata can be stored in a plaintext `.meta.json` sidecar file alongside the Age-encrypted `.age` entry:

```
sec/lockbox/entries/
├── postgres/
│   ├── test.age           # Age-encrypted password
│   └── test.meta.json     # Plaintext connection metadata
```

The sidecar JSON schema:

```json
{
  "driver": "postgres",
  "username": "rivers",
  "hosts": ["192.168.2.209:5432", "192.168.2.210:5432"],
  "database": "rivers",
  "options": { "authSource": "admin", "replicaSet": "rs0" }
}
```

| Field | Type | Description |
|---|---|---|
| `driver` | string | Driver name matching a registered driver |
| `username` | string | Database username or bind DN |
| `hosts` | string[] | `"host:port"` entries; first is primary, rest are replicas/cluster nodes |
| `database` | string | Database, bucket, or base DN |
| `options` | object | Driver-specific key-value options (optional) |

Sidecars are never encrypted — they contain no secrets. The `TestCredentials` helper in `rivers-core/tests/common/mod.rs` combines the decrypted password from `.age` with metadata from `.meta.json` to build a complete `ConnectionParams`.

### 3.4 Naming rules

- Name must match `[a-z][a-z0-9_/.-]*` — lowercase alphanumeric plus `_`, `/`, `.`, `-`; starting with a letter
- Maximum length: 128 characters
- Forward-slash segments are for organizational grouping only (`postgres/orders-prod`) — no hierarchy is enforced
- Names are immutable after creation — use `rivers lockbox alias` to add an alias, `rivers lockbox rotate` to update the value

### 3.4 Namespace uniqueness

Names and aliases share a single namespace. No two entries may have the same name, and no entry may have an alias that matches any other entry's name or alias. The CLI enforces this at write time. `riversd` enforces it again at startup — duplicates cause a startup failure.

## 4. Alias Resolution

An alias is an alternative name for an entry. Aliases allow:

- Shorthand references (`anthropic_key` instead of `anthropic/api_key`)
- Stable references across key rotation (`llm_key` always points to the current LLM key regardless of which provider is active)
- Environment-neutral references (`db_password` in config, actual entry varies per environment)

Resolution is flat — aliases resolve directly to entries, not to other aliases. Alias chains are not supported. CLI validation rejects circular or chained alias references at write time.

### Resolution order

1. Exact match on `name`
2. Exact match on any `alias`
3. Not found — `LockBoxError::EntryNotFound` — startup fails

Both steps happen against the in-memory entry index. Resolution is O(1) via hash map built at startup (keyed by both names and aliases, mapping to entry positions — not secret values). Secret values are fetched from disk on demand via `fetch_secret_value()`. <!-- SHAPE-5 amendment: index only, values from disk per-access -->

```
lockbox://postgres/orders-prod     → name match  → entry value
lockbox://orders-db                → alias match → same entry value
lockbox://missing_key              → no match    → startup failure
```

## 5. riversd.conf Integration

The keystore file and key source are declared in `riversd.conf`:

```toml
[lockbox]
path        = "/etc/rivers/riversd.rkeystore"   # required; must be absolute path
key_source  = "file"                             # "env" | "file" | "agent"
key_file    = "/etc/rivers/lockbox.key"          # used when key_source = "file"
```

`path` must be an absolute path. Relative paths are rejected at config validation:

```
RiversError::Config: lockbox.path must be an absolute path, got "riversd.rkeystore"
```

If no `lockbox://` reference exists anywhere in the config, the `[lockbox]` section is optional and the keystore file is not loaded. If any `lockbox://` reference exists and `[lockbox]` is not configured, startup fails:

```
RiversError::Config: lockbox reference found but [lockbox] is not configured
  → add [lockbox] section to riversd.conf
```

### Complete riversd.conf example

```toml
[base]
name = "orders-api"
host = "0.0.0.0"
port = 8080

[lockbox]
path       = "/etc/rivers/riversd.rkeystore"
key_source = "file"
key_file   = "/etc/rivers/lockbox.key"

[data.datasources.primary_db]
driver             = "postgresql"
host               = "db.internal"
port               = 5432
database           = "orders"
credentials_source = "lockbox://postgres/orders-prod"

[data.datasources.cache]
driver             = "redis"
host               = "cache.internal"
port               = 6379
credentials_source = "lockbox://redis/prod"

[data.datasources.anthropic]
driver             = "http"
base_url           = "https://api.anthropic.com"
auth               = "bearer"
credentials_source = "lockbox://anthropic/api_key"
```

## 6. Decryption Key Sources

The Age identity (decryption key) is never stored alongside the keystore file. Three sources are supported via `key_source` in `[lockbox]`:

### `key_source = "env"` (default for development)

```toml
[lockbox]
key_source  = "env"
key_env_var = "RIVERS_LOCKBOX_KEY"   # default var name; override with key_env_var
```

`RIVERS_LOCKBOX_KEY` contains the Age identity string (X25519 private key in Bech32 encoding, `AGE-SECRET-KEY-1...`). Rivers reads it at startup, decrypts the keystore, then clears the variable from the process environment.

### `key_source = "file"` (for production deployments)

```toml
[lockbox]
key_source = "file"
key_file   = "/etc/rivers/lockbox.key"   # absolute path, Age identity file
```

The identity file contains the Age secret key. File permissions are validated at startup — if the file is group- or world-readable, startup fails:

```
RiversError::Security: lockbox key file /etc/rivers/lockbox.key has insecure permissions (mode 0644)
  → chmod 0600 /etc/rivers/lockbox.key
```

### `key_source = "agent"` (for hardware key / SSH agent)

```toml
[lockbox]
key_source     = "agent"
agent_socket   = "/run/user/1000/gnupg/S.gpg-agent.ssh"  # SSH_AUTH_SOCK
recipient_file = "/etc/rivers/lockbox.pub"                 # Age recipient (public key)
```

Rivers uses the SSH agent via the `age-ssh` plugin. The keystore is encrypted to an SSH key held in the agent. Useful for hardware security keys (YubiKey, etc.) or when the operator requires physical presence for decryption.

### File permissions enforcement

| File | Required mode | Error if violated |
|---|---|---|
| `.rkeystore` | `600` | `LockBoxError::InsecureFilePermissions` |
| `key_file` (when `key_source = "file"`) | `600` | `LockBoxError::InsecureFilePermissions` |

`riversd` and `rivers lockbox` both enforce these permissions before attempting to read either file. `rivers lockbox init` sets `600` on both files automatically.

## 7. rivers lockbox CLI

All keystore operations go through the `rivers lockbox` subcommand. The keystore file is never edited directly. All commands that modify the keystore: read the existing file → decrypt → apply change → re-encrypt → write back atomically (write to `.tmp`, then rename).

The CLI reads key source config from `riversd.conf` unless overridden by `--config <path>`.

### `rivers lockbox init`

Generate a new Age keypair and an empty encrypted keystore.

```bash
rivers lockbox init \
    --path /etc/rivers/riversd.rkeystore \
    --key-source env
# Generates a new Age keypair
# Prints: AGE-SECRET-KEY-1... (store this in RIVERS_LOCKBOX_KEY)
# Prints: age1...             (recipient public key)
# Creates empty keystore at --path
# Sets permissions 600 on keystore file
```

Refuses if file already exists. Use `--force` to overwrite.

---

### `rivers lockbox add`

Add a new entry. Prompts for value on stdin — never as a flag, to prevent value appearing in shell history or process list.

```bash
rivers lockbox add \
    --name anthropic/api_key \
    --type string \
    --alias llm_key \
    --alias anthropic_key
# Value: **** (hidden input)
# ✓ Added "anthropic/api_key" with 2 aliases
```

Exits non-zero if `name` or any alias already exists in the keystore.

---

### `rivers lockbox list`

List all entries — names, types, aliases, updated timestamps. Values are never printed.

```bash
rivers lockbox list

# NAME                   TYPE      ALIASES                      UPDATED
# anthropic/api_key      string    anthropic_key, llm_key       2026-01-15
# postgres/orders-prod   string    db/orders, orders-db         2026-01-15
# jwt_signing_key        base64url jwt_key, signing_key         2026-01-15
```

---

### `rivers lockbox show`

Display an entry's value. Requires explicit confirmation — not scriptable without `--yes` (which emits a warning log).

```bash
rivers lockbox show --name anthropic/api_key
# WARNING: This will display a secret value. Continue? [y/N] y
# name:    anthropic/api_key
# type:    string
# aliases: anthropic_key, llm_key
# value:   sk-ant-...
```

---

### `rivers lockbox alias`

Add one or more aliases to an existing entry.

```bash
rivers lockbox alias --name anthropic/api_key --alias claude_key
# ✓ Alias "claude_key" added to "anthropic/api_key"
```

Exits non-zero if the alias already exists under any entry.

---

### `rivers lockbox unalias`

Remove an alias from an entry.

```bash
rivers lockbox unalias --name anthropic/api_key --alias old_alias
# ✓ Alias "old_alias" removed from "anthropic/api_key"
```

---

### `rivers lockbox rotate`

Replace an entry's value. Name and aliases are preserved. Prompts for new value on stdin.

```bash
rivers lockbox rotate --name anthropic/api_key
# Current value will be replaced. Continue? [y/N] y
# New value: **** (hidden input)
# ✓ Rotated "anthropic/api_key" — updated timestamp set
# → new connections will pick up the updated credential automatically
```

<!-- SHAPE-5 amendment: no restart required -->
No restart is required. Because secret values are read from disk on every access, new connections will use the updated credential automatically. Existing connections cycle via `max_lifetime` or health check failure.

---

### `rivers lockbox remove`

Remove an entry and all its aliases. Requires confirmation.

```bash
rivers lockbox remove --name postgres/orders-prod
# WARNING: Removing "postgres/orders-prod" will break any lockbox:// references to this entry.
# Aliases removed: db/orders, orders-db
# Type entry name to confirm: postgres/orders-prod
# ✓ Removed
```

`--yes` flag skips confirmation. Exits non-zero if name not found.

---

### `rivers lockbox rekey`

Re-encrypt the entire keystore with a new Age identity. Used for identity rotation.

```bash
rivers lockbox rekey
# Generates new Age keypair
# Re-encrypts all entries under new key
# Prints new AGE-SECRET-KEY-1...
# Old key is no longer valid for this keystore
# → Update riversd.conf [lockbox] key source to use new key
# → Restart riversd
```

The old key file is not deleted — the operator is responsible for retiring it.

---

### `rivers lockbox validate`

Verify the keystore is well-formed, all entries are readable, and all `lockbox://` references in the config resolve. Non-destructive.

```bash
rivers lockbox validate --config /etc/rivers/riversd.conf
# ✓ Keystore decrypted successfully
# ✓ 3 entries loaded
# ✓ No duplicate names or aliases
# ✓ All lockbox:// references resolve
# Keystore is valid.
```

Exits non-zero on any error — permission issue, decryption failure, duplicate names, unresolved references.

### Exit codes

| Code | Meaning |
|---|---|
| 0 | Success |
| 1 | Entry not found |
| 2 | Entry already exists (add conflict) |
| 3 | Decryption failed (wrong key, corrupted file) |
| 4 | Permission error (file permissions, read/write failure) |
| 5 | Keystore validation error (duplicate names, malformed TOML) |
| 6 | User cancelled (confirmation prompt declined) |

## 8. riversd Startup Resolution

### 8.1 Startup sequence

LockBox resolution occurs during `riversd` startup, after config is parsed and before any datasource connections are initialized:

```
riversd startup
    │
    ├─ 1. parse riversd.conf
    ├─ 2. collect all lockbox:// URIs from datasource configs
    │
    ├─ 3. [lockbox] section present?
    │       NO and URIs exist → startup failure: LockBoxConfig missing
    │       NO and no URIs   → LockBox inactive, continue
    │       YES              → continue
    │
    ├─ 4. check file permissions on .rkeystore and key source file
    │       failure → LockBoxError::InsecureFilePermissions
    │
    ├─ 5. resolve key source (read env var / read key file / connect agent)
    │       failure → LockBoxError::KeySourceUnavailable
    │
    ├─ 6. decrypt .rkeystore
    │       failure → LockBoxError::DecryptionFailed
    │
    ├─ 7. parse decrypted TOML
    │       failure → LockBoxError::MalformedKeystore
    │
    ├─ 8. validate entries (no duplicate names/aliases, naming rules)
    │       failure → LockBoxError::DuplicateEntry | InvalidEntryName
    │
    ├─ 9. build in-memory name+alias → entry index (no secret values stored)
    │         <!-- SHAPE-5 amendment: index only, not value map -->
    │
    ├─ 10. resolve all lockbox:// URIs from step 2 (verify entries exist by index)
    │        any unresolved → LockBoxError::EntryNotFound (with datasource context)
    │
    ├─ 11. zeroize plaintext TOML bytes and decrypted Age bytes
    │
    └─ 12. initialize datasource connection pools
    │        each pool calls fetch_secret_value(index) → decrypt from disk, use, zeroize
```

### 8.2 Failure behavior

Any failure in steps 3–10 causes immediate startup failure. `riversd` exits non-zero with a structured error message identifying the failure type, file path, and (where applicable) the unresolved URI and datasource name. Partial startup is not permitted — `riversd` does not serve traffic with unresolved credentials.

### 8.3 Runtime credential access

After startup, credentials are accessed by the datasource connection pool manager via an internal `LockBoxResolver::fetch_secret_value(uri)` call. This reads the entry from disk, decrypts it, returns the value, and the caller zeroizes it after use. No secret values are cached in memory. This is host-side Rust code only — not accessible from ProcessPool isolates. <!-- SHAPE-5 amendment: on-demand decryption via fetch_secret_value() -->

### 8.4 Credential rotation

<!-- SHAPE-5 amendment: no restart required for rotation -->
`rivers lockbox rotate` writes the new credential value to disk (re-encrypts the keystore). No `riversd` restart is required. Because secret values are read from disk on every access (not cached in memory), the next connection established by the pool will automatically use the new credential. Existing connections continue with their current credentials until they are recycled via `max_lifetime` or health check failure. The `CredentialRotated` event is not used — connection cycling handles propagation naturally.

## 9. Datasource Integration

### 9.1 credentials_source field

All driver datasource configs accept a `credentials_source` field. In v1, the only valid value format is `lockbox://{name-or-alias}`.

```toml
[data.datasources.primary_db]
driver             = "postgresql"
host               = "db.internal"
port               = 5432
database           = "orders"
credentials_source = "lockbox://postgres/orders-prod"
```

### 9.2 Credential value semantics by driver

The `value` field in a keystore entry is a raw string. The driver is responsible for interpreting it:

| Driver | Expected credential format |
|---|---|
| `postgresql` | Full connection string `postgresql://user:pass@host:port/db` or `user:password` |
| `mysql` | Full connection string `mysql://user:pass@host:port/db` or `user:password` |
| `redis` | Password string or full `redis://:password@host:port` |
| `mongodb` | Full URI `mongodb://user:pass@host:port` |
| `http` (bearer) | Token string — passed as `Authorization: Bearer {value}` |
| `http` (api_key) | Key string — passed as configured `auth_header: {value}` |
| `http` (basic) | `username:password` — base64-encoded by driver |
| `kafka` | `username:password` — used for SASL authentication |
| `elasticsearch` | `username:password` or API key string |

If the credential format is invalid for the driver, the datasource connection pool fails to initialize at startup — not at first query.

### 9.3 Credential-free datasources

`credentials_source` is optional. Datasources that do not require authentication omit the field entirely. In `resources.toml`, declare `nopassword = true` to indicate no lockbox entry is needed:

```toml
# resources.toml
[[datasources]]
name       = "contacts"
driver     = "faker"
nopassword = true
required   = true
```

```toml
# app.toml — no credentials_source
[data.datasources.contacts]
driver     = "faker"
nopassword = true
```

## 10. Handler Integration

Handler code does not interact with LockBox directly. Credentials are a datasource concern — resolved at the host layer and delivered to the driver. The isolate receives an opaque token for the datasource, not the credential value.

```typescript
// Handler — no lockbox reference, no credential handling
export async function generate(req: Rivers.Request) {
    // "anthropic" datasource has credentials resolved host-side from lockbox
    for await (const chunk of Rivers.view.stream("generate_completion", {
        prompt: req.body.prompt,
    })) {
        yield chunk;
    }
}
```

```toml
# Datasource config — lockbox reference is here, not in handler
[data.datasources.anthropic]
driver             = "http"
base_url           = "https://api.anthropic.com"
auth               = "bearer"
credentials_source = "lockbox://anthropic/api_key"
```

The `lockbox://` URI appears only in `riversd.conf` — in datasource configs, session signing key config, and other infrastructure config. It never appears in TypeScript, JavaScript, or WASM handler source.

## 11. Security Model

### At rest

- Keystore file is Age-encrypted (X25519 + ChaCha20-Poly1305 AEAD)
- Plaintext never written to disk — decrypted in memory at startup only
- File permissions validated at startup and enforced by CLI on write — `600` required

### In memory

- Plaintext TOML cleared after parsing — only the entry index (name → entry position) is retained, not secret values <!-- SHAPE-5 amendment -->
- Entry index lives in the `riversd` host process — not in any isolate
- Secret values are decrypted from disk per-access, used, and immediately zeroized
- Isolates hold opaque datasource tokens, not credential values

### In transit

- Credentials never logged — errors log URI and datasource name only, never credential values
- Credentials never appear in error messages
- `rivers lockbox show` requires explicit terminal confirmation

### Threat model

| Threat | Mitigation |
|---|---|
| Keystore file read by unauthorized user | `600` permissions enforced at startup and by CLI on write |
| Key file read by unauthorized user | `600` permissions enforced at startup |
| Credential value in shell history | `add` and `rotate` read value from stdin — never as a CLI argument |
| Credential value in process list | Same — stdin only, never `--value` flag |
| Credential leaking into ProcessPool isolate | Host-side resolution only. No `Rivers.lockbox` API in v1. Opaque token model. |
| Plaintext keystore left in memory after parsing | Plaintext TOML and Age-decrypted bytes zeroized after entry index is built; no secret values retained in memory — decrypted per-access and zeroized immediately <!-- SHAPE-5 amendment --> |
| Credential in log output | `riversd` never logs credential values |
| Keystore corruption | `rivers lockbox validate` — run after any manual file operation |

### Limitations

- **Root access** — a root user can read the key file and decrypt the keystore. LockBox protects against unprivileged access, not root compromise.
- **Process memory inspection** — resolved credential values are held in process memory. A sufficiently privileged attacker with access to the `riversd` process memory can extract them. This is an inherent limitation of any in-process secret store.
- **Credential rotation propagation** — rotation does not require a restart, but existing connections continue with their current credentials until recycled via `max_lifetime` or health check failure. There is no instant propagation to all active connections. <!-- SHAPE-5 amendment -->

## 12. Validation Rules

Enforced at `riversd` startup and by `rivers lockbox` at write time:

| Rule | Error |
|---|---|
| `[lockbox]` absent, `lockbox://` URIs present | `LockBoxError::ConfigMissing` |
| `lockbox.path` file not found | `LockBoxError::KeystoreNotFound: {path}` |
| `lockbox.path` not an absolute path | `RiversError::Config: lockbox.path must be absolute` |
| `.rkeystore` permissions not `600` | `LockBoxError::InsecureFilePermissions: {path}` |
| `key_file` permissions not `600` (when `key_source = "file"`) | `LockBoxError::InsecureFilePermissions: {path}` |
| Decryption failure | `LockBoxError::DecryptionFailed — check key source matches keystore` |
| Malformed TOML payload | `LockBoxError::MalformedKeystore: {parse error}` |
| Duplicate entry name or alias | `LockBoxError::DuplicateEntry: "{name}" appears in multiple entries` |
| Unresolved `lockbox://` URI | `LockBoxError::EntryNotFound: "{uri}" referenced by datasource "{name}"` |
| Entry name fails naming rules | `LockBoxError::InvalidEntryName: "{name}" — must match [a-z][a-z0-9_/.-]*` |
| Entry name exceeds 128 characters | `LockBoxError::InvalidEntryName: "{name}" — maximum length 128 characters` |
| `rivers lockbox add` with existing name | `Entry "{name}" already exists — use rivers lockbox rotate to update` |
| `rivers lockbox alias` with existing alias | `Alias "{alias}" already exists under entry "{name}"` |

---

## 13. v2 Upgrade Path

In v2, RPS replaces the local resolver for multi-node deployments. The upgrade path:

1. Import existing keystore into RPS:
   ```bash
   rivers rps import-keystore --from /etc/rivers/riversd.rkeystore
   ```

2. Update `riversd.conf` — replace `[lockbox]` with `[rps]`:
   ```toml
   # v1 config
   [lockbox]
   path       = "/etc/rivers/riversd.rkeystore"
   key_source = "env"

   # v2 config — replace [lockbox] with [rps]
   [rps]
   endpoint     = "https://rps.internal:7443"
   trust_bundle = "/etc/rivers/trust-bundle.age"
   ```

3. Remove `RIVERS_LOCKBOX_KEY` env var from deployment (if `key_source = "env"` was used)

No handler code changes. No datasource config changes. All `lockbox://` URIs in `riversd.conf` remain valid — RPS resolves them identically to the local resolver.

Single-node deployments may continue using the local keystore indefinitely — RPS is not required for single-node operation in v2.
